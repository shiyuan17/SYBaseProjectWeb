# ROUTER_RULES.md — 路由设计规范

## 目标与适用范围

本文件定义 `Vue Router` 的路径设计、命名策略、守卫边界与导航元信息规则。

- 适用对象：前端开发者、权限与导航维护者、AI 助手
- 适用范围：路由表、路由守卫、菜单映射、面包屑与标题处理
- 规范定位：统一“路由怎么命名、怎么拆、守卫里该做什么”

## 强制规则

### 1. 路径与命名

- 路由路径使用英文小写短横线风格
- 路由名称必须唯一、稳定、可读
- 菜单型页面必须定义明确的路由元信息，例如标题、权限码、缓存策略、菜单归属
- 禁止在页面或组件里散落硬编码路由路径和路由名

### 2. 路由组织

- 路由注册放在 `src/router`
- 业务页面懒加载，不在路由文件里直接实现业务逻辑
- 复杂模块的路由配置可按业务域拆分后统一汇总
- 页面布局、导航壳、异常页、登录页应作为明确的路由层概念管理

### 3. 守卫边界

- 守卫只处理导航准入、登录态校验、权限判断、标题更新、基础前置条件
- 守卫中不得塞入业务请求编排、页面数据初始化或复杂 UI 逻辑
- 权限失败、未登录、无效路由必须有明确重定向或错误页策略

### 4. 参数与查询

- 路由参数必须有清晰语义和类型约束
- 查询参数用于表达筛选、分页、排序、标签等可分享状态
- 需要深链接恢复的页面状态应通过查询参数或路由状态显式表达

### 5. 面包屑与标题

- 页面标题和面包屑生成规则必须统一
- 不得在多个页面各自手写标题更新逻辑
- 动态标题或面包屑依赖异步数据时，必须定义回退文案策略

## 推荐实践

- 用常量或 helper 统一管理常用路由名
- 将权限码、标题、缓存、菜单显示规则收敛到 `meta`
- 对高频导航链路补充进入前校验和异常回退设计
- 通过模块化路由配置让业务边界和目录结构保持一致

## 反例/禁用项

- 使用 `'/list'`、`'/detail'` 这类脱离业务语境的顶层路径
- 在路由守卫中发起大量业务请求，再把结果写入多个 store
- 页面中到处手写 `router.push('/xxx')`
- 无标题、无权限元信息、无异常跳转约定的随意路由配置

## 检查清单

- [ ] 路径、名称、元信息符合统一风格
- [ ] 路由配置与业务模块边界一致
- [ ] 守卫只处理导航与权限，不承载业务请求
- [ ] 参数、查询、深链接策略清晰
- [ ] 标题和面包屑规则统一
