本报告基于前端仓库 SYBaseProjectWeb 与同级后端仓库 SYBaseProject 的静态源码、治理文档、路由、API service、后端 Controller 和数据库迁移进行梳理。它用于帮助团队建立当前系统全貌、识别业务链路和状态边界，并沉淀后续治理风险。

说明：本报告不代表生产环境实时健康状态；未运行完整 dead-code、性能压测或接口契约自动比对，因此风险章节将“确认问题”和“候选风险信号”分开表述。

02 系统整体架构

系统整体架构

当前系统是“前端 Vben 管理端 + 后端 Spring Boot 多中心服务 + 达梦数据库 + 外部系统网关”的业务平台。前端负责模块化页面、权限菜单、交互工作台和 API 适配；后端以 auth-center、user-center、bl-center 分离认证、用户和业务能力；数据库通过 Flyway 迁移维护业务表和兼容性演进。

前端架构

前端仓库是 Vben Admin monorepo，主应用位于 apps/web-ele。启动链路为 main.ts 初始化偏好命名空间和兼容迁移，再动态加载 bootstrap.ts 注册组件适配器、表单适配器、Element Plus loading、Vben loading、i18n、Pinia、权限指令、Tippy、Router、Motion 和动态标题。

层次 现状 说明
应用壳 main.ts、bootstrap.ts、Vben packages 统一承载偏好、样式、布局、指令、路由、国际化和 store 初始化。
业务模块 src/modules/<domain> 按 M1-M6、通知、看板等领域组织 views、components、api、types、utils。
请求层 src/api/request.ts 统一 baseURL、Bearer token、语言头、响应 envelope 解包、未认证处理和全局错误提示。
状态层 Vben stores + src/store/auth.ts + 少量模块 store 认证流程由本地 auth store 编排，通知中心有独立 store；多数页面状态位于组件或 composable。
路由层 src/router/routes/modules 每个业务域有独立 route module，meta 中包含权限码、图标、标题、菜单隐藏策略。

后端架构

后端同级仓库 SYBaseProject 是 Java 17 与 Spring Boot 3 Maven 多模块工程。业务核心在 bl-center，公共能力在 common-core、common-security、common-web，认证和用户能力分别位于 auth-center 与 user-center。

后端模块 职责 关键技术/约束
auth-center 认证、登录、令牌与会话相关能力。 前端开发代理将 /api/v1/auth 指向 8081。
user-center 用户、角色、权限基础能力。 与菜单、权限码、系统管理页面共同组成鉴权基础。
bl-center M1-M6 业务 API、流程编排、统计、档案、通知、外部集成网关。 Spring Web、Validation、Actuator、Cache、JDBC、MyBatis Plus、Flyway、达梦驱动。
common/* 核心模型、安全、Web、测试公共能力。 约束跨模块复用边界。
tools/app-cli 工程工具或命令行辅助。 不在前端运行链路内。

数据库

第三方服务与外部系统

后端以 Gateway 抽象对接外部系统，目前静态源码中可见临床申请、计费、历史报告、标签打印、技术标记等网关，同时存在对应 Stub 实现。监控侧包含 Prometheus、Loki、Grafana、Alloy 等线索；Redis、MQ、Elasticsearch、AI、storage 等基础设施目录有占位或预留迹象。

认证与访问流程

用户登录后，前端保存 access token，并并行获取用户信息与 access codes。后续请求统一携带 Bearer token 和语言头；认证失败时清除 token 或进入登录过期流程。菜单由后端返回并与前端 fallback/component definition 对齐。 

sequenceDiagram
  participant User as 用户
  participant Web as web-ele
  participant AuthStore as auth store
  participant Auth as auth-center
  participant BL as bl/user API
  participant Router as Vue Router

  User->>Web: 输入账号密码
  Web->>AuthStore: authLogin(params)
  AuthStore->>Auth: POST /v1/auth/login
  Auth-->>AuthStore: accessToken
  par 获取用户信息
    AuthStore->>Auth: GET /v1/auth/me
  and 获取权限码
    AuthStore->>Auth: GET /v1/auth/access-codes
  end
  AuthStore->>Web: 写入 user/accessCodes/token
  Web->>BL: GET /v1/menus
  BL-->>Web: 菜单树
  Web->>Router: 导航到默认首页或 redirect

08 风险分析

风险分析

以下风险分为确认风险、静态候选风险和治理建议。确认风险来自本次已核对的源码事实；候选风险需要后续专项 dead-code、调用关系、契约测试或运行时验证进一步确认。

确认风险 病理接收路由重复注册 /workflow/pathology-receipt 和 PathologyReceipt 同时出现在 M2 与 M3 route module，需确认 Vue Router 当前合并策略和菜单来源，避免名称冲突、keep-alive 或权限判断不一致。
确认风险 占位页面进入菜单 危化品、医疗废物路由已存在但指向 MenuPlaceholderView。如果权限已开放，用户会看到未接入页面。
候选风险 外部网关仍为 Stub 临床申请、计费、历史报告、标签打印、技术标记均有 Stub 实现。上线前需明确哪些环境使用真实适配器。
候选风险 历史兼容迁移较多 多处 relax、reconcile、ensure 迁移说明数据库历史兼容压力大，状态字段和 nullable 字段需重点回归。

风险点 证据 影响 建议
占位功能页 /operation-resources/hazardous-chemicals、/operation-resources/medical-waste 菜单可见但业务不可用，容易形成验收缺口。 在需求、菜单权限或发布说明中明确“预留入口”；未接入前限制角色可见性。

  • 前端统一使用 RequestClient，默认 responseReturn: data。
  • 响应 envelope 以 code、data、SUCCESS 解包。
  • 刷新 token 被禁用，认证失败执行本地 token 清理或登录过期弹窗/跳转。
  • 开发代理将 /api/v1/auth 转到 8081，其余 /api/v1 转到 8080。
  • 下载使用 requestClient.download 或 body response，用于 CSV/导出类场景。

持久化模型特点

  • 业务主表与流程事件并存，例如 M2 使用 workflow_events 补充追踪。
  • 多模块共享状态词，但状态字段分散在不同表内，适合后续建立统一状态词典。
  • 兼容迁移显示历史数据存在缺列、非空约束不兼容或唯一约束变化，需要以样本数据做回归。
  • M6 统计表与业务主表分离，支持模板、指标和导出任务，但统计口径需要明确数据来源。
  • 通知中心采用数据库表落地，不依赖已确认的运行时 MQ 主链路。
10 系统现状总结

系统现状总结

完整度 业务主链路已成型 M1-M6、通知中心、统计和运营支持均有前端页面、后端 API 和数据库结构支撑，病理全流程雏形完整。
工程化 前后端边界清晰 前端按业务模块组织，后端按多中心和 DDD 分层组织，统一请求客户端和 Controller/API 边界较清楚。
演进压力 历史兼容痕迹明显 数据库迁移和兼容路由较多，说明系统在快速迭代中持续修正字段、约束和入口。
重点风险 状态与权限需治理 跨模块状态码、路由/菜单/权限、外部 Stub、敏感数据导出是后续上线前应优先收敛的风险。

总体判断：当前系统已经从“页面原型”进入“业务流程平台”阶段，核心业务对象和跨模块链路都已具备实现基础。M2 与 M3 是最复杂、最需要治理的模块；M4 涉及报告和患者敏感数据，是权限、审计与版本一致性的重点；M5 与 M6 正在从支撑能力向正式运营能力扩展。后续建议优先处理路由重复、占位入口、外部 Stub readiness、状态词典和权限矩阵自动校验，再逐步治理大型模块和历史迁移债务。

建议优先级

  1. 修正或确认 PathologyReceipt 重复 route name/path 的设计意图，并补充路由合并测试。
  2. 建立前后端权限矩阵，核对 route meta、菜单、component definitions 和后端 @RequirePermission。
  3. 为核心对象建立状态词典：标本、技术任务、包埋、报告、医嘱、库存、集成任务。
  4. 为外部 Gateway 建立环境 readiness 清单，区分 Stub、联调、生产适配器。
  5. 对 M2/M3 大模块做专项复杂度和重复逻辑扫描，先抽离稳定 mapper、状态映射和测试夹具。
