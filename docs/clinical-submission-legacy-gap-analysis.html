Legacy Gap Analysis

临床送检现状 vs 旧工作站参考 HTML 对齐分析

本文档以当前仓库 specimen-workflow 模块为现状真值，以参考 clinical-pathology-workstation-analysis.html 中描述的旧临床病理工作站为对照基线， 聚焦“当前前端已经承接了什么、哪些能力只是合并呈现、哪些旧模块仍未独立落地”，并给出分阶段执行方案。

保留 4 个可见入口 不回退为 11 个独立菜单 当前实现优先于历史概念 后端主链路仅覆盖 M2 主流程
4 / 11 当前以 4 个可见入口承载旧工作站 11 个业务模块。
3 类结论 差异统一归类为“已合并承载 / 部分覆盖 / 当前缺失或需扩展”。
P0 - P3 执行路线按文档补齐、前端补齐、前端增强、后端扩展四阶段推进。

流程对比总览

旧参考 HTML 把临床病理工作站描述为 11 个模块的桌面式工作站；当前 web 端则将临床送检收敛为 4 个可见入口，并通过隐藏兼容路由保留老入口别名。

申请与登记

当前承接旧“标本登记”主链路

  • 入口：/workflow/submission-registration
  • 承载：申请单列表、手工建单、第三方导入、标本登记、标签补打
  • 兼容：/workflow/application-list、/workflow/specimen-management、/workflow/clinical-register
固定与转运

当前承接旧“固定/转运相关”链路

  • 入口：/workflow/fixation-transport
  • 页签：固定核对、转运交接
  • 兼容：/workflow/fixation-verify、/workflow/transport-handover
病理接收

当前把旧“接收相邻环节”提升为显式入口

  • 入口：/workflow/pathology-receipt
  • 承载：按转运单接收、条码直收、质控、退回、拒收
  • 兼容：/workflow/specimen-receipt
追踪与异常

当前承接旧“综合信息 + 追踪查询”

  • 入口：/workflow/tracking-exception
  • 页签：申请单列表、标本列表
  • 兼容：/workflow/tracking-query
核心判断 当前 web 端不是“缺少临床送检流程”，而是采用了“按连续操作链收敛入口”的产品结构。因此本轮对齐不应回退菜单模型，而应先把旧模块映射与缺口描述讲清。

关键差异定性

已合并承载
  • 旧“标本登记”已拆分到 ApplicationManageDialog、SpecimenRegisterDialog、ApplicationListView、SpecimenManagementView。
  • 旧“固定 / 转运”已被 FixationTransportView 的两个页签合并承载。
  • 旧“综合信息”与“追踪查询”已被 TrackingApplicationListView 与 TrackingSpecimenListView 合并承载。
部分覆盖
  • 旧“离体时间”目前只作为申请单字段与追踪详情展示，不是独立工作台。
  • 旧“标本管理”目前更偏列表、详情、补打标签、固定核验，不等同旧桌面端异常修正工作站。
  • 旧“接收后病例 / 病理号生成”当前有结果回显，但完全依赖接收接口返回，不是前端可控流程节点。
  • 旧“标本核对”与旧“标本出库”只被当前流程部分吸收，语义并不完全等价。
当前缺失或未独立呈现
  • 标本核对独立页
  • 标本核对记录
  • 条码绑定独立页
  • 离体时间设置独立页
  • 标本确认独立页
  • 标本入库 / 出库独立页
  • 申请单打印、检查状态、已签发报告查看、患者核对开关
需要写死给实现者的判断 以上缺口里，除了文案、入口说明、字段可见性和上下文透传之外，绝大多数都不是“前端多写一个页面”就能完成。它们往往意味着后端要同时补接口、状态机、权限种子和测试。

后端联动边界

当前后端 M2 主链路是明确可验证的，但它只覆盖“申请 -> 登记 -> 固定 -> 转运 -> 接收 -> 追踪”，并没有为旧 11 模块中的全部独立工作站提供一一对应的领域能力。

当前已落的主链路

  • POST /api/v1/applications：创建申请单
  • POST /api/v1/clinical-applications/import：第三方导入
  • POST /api/v1/specimens/register：标本登记
  • POST /api/v1/specimen-fixations/start / complete：固定
  • POST /api/v1/transport-orders / print / handover：转运
  • POST /api/v1/specimen-receipts / by-barcodes：接收
  • GET /api/v1/applications/{id}/tracking / GET /api/v1/specimens/barcodes/{barcode}/tracking：追踪

当前不足以 1:1 支撑的旧模块

  • 独立标本核对状态流与核对记录批次查询
  • 条码绑定 / 改绑 / 修正
  • 标本确认状态节点
  • 病理科入库 / 出库台账
  • 旧工作站中的申请单打印、检查状态、报告查看、患者核对配置开关
边界结论 如果后续要把旧 11 模块“逐个补成页面”，那已经不是单纯的前端对齐，而是新的领域扩展任务；必须同步变更接口、权限、菜单种子、状态机和测试。

执行路线

P0

文档补齐

  • 新增本对齐分析 HTML。
  • 把“旧 11 模块 -> 当前 4 入口”的映射写死。
  • 把“已合并承载 / 部分覆盖 / 当前缺失”写成固定判断。
P1

仅前端可做的对齐

  • 补文案和映射说明。
  • 补兼容入口关系说明。
  • 补详情里的旧字段可见性与语义提示。
  • 补从列表到追踪 / 登记 / 接收的上下文透传与回跳说明。
P2

前端增强但不改权限模型

  • 在现有 4 入口内增加“旧模块映射视图”或“工作台分区”。
  • 把离体时间、核对语义、综合台账展示得更接近旧流程。
  • 保持现有权限与路由结构不变。
P3

需人工确认后的领域扩展

  • 条码绑定、核对记录、确认、入库、出库等独立模块。
  • 新增菜单、权限种子、接口、状态机与测试。
  • 属于黄区 / 红区交界，必须先人工确认再实施。

验收与校验建议

P0 文档验收

  • 11 个旧模块全部出现在映射表中。
  • 每个旧模块都被标记为“已合并 / 部分覆盖 / 缺失”之一。
  • 每个现状结论都能回指到具体前端视图、组件、路由或后端接口 / 测试。

后续实现阶段验收

  • 兼容路由继续覆盖 submission-registration、fixation-transport、pathology-receipt、tracking-exception 及隐藏重定向路由。
  • 现有 specimen-workflow 视图测试继续通过。
  • 后端至少复跑 M2 角色场景集成测试，确认主链路无回归。

最终结论

当前 web 端临床送检并不是“少了一整套流程”，而是采用了与旧桌面工作站完全不同的信息架构：把旧 11 模块压缩为 4 个连续工作流入口，再用弹窗、详情抽屉和兼容路由承接上下文。真正需要做的不是盲目“恢复 11 菜单”，而是先把旧模块映射讲清，再分辨哪些只是表达方式差异，哪些才是确实缺少的领域能力。

建议落地方向 先按 P0 / P1 完成文档和前端表达层对齐；只有在业务明确要求回补旧工作站能力时，才进入 P3 的接口与状态机扩展。

文档假设：“当前临床送检”专指本仓库 specimen-workflow 模块实现，不代表线上运行态，也不默认等价于历史桌面端。
