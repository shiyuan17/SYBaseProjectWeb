# STATE_RULES.md — Pinia 状态管理规范

## 目标与适用范围

本文件定义 `Pinia` 在项目中的使用边界、状态分层与 store 设计规则。

- 适用对象：前端开发者、模块维护者、AI 助手
- 适用范围：全局 store、模块 store、与 store 相关的组合式函数
- 规范定位：统一“什么应该进入 store，什么不应该进入 store”

## 强制规则

### 1. 状态分层

- 全局稳定状态放在 `apps/web-ele/src/store`，例如用户会话、权限、布局、主题
- 业务模块状态优先放在 `src/modules/<domain>/store`
- 组件局部临时状态优先留在组件内部，不得无理由上提到 Pinia

### 2. Store 职责

- `state` 负责稳定可观察数据
- `getter` 负责派生状态与只读聚合
- `action` 负责异步流程、状态变更编排与副作用封装
- store 不得直接承载视图层一次性临时态，例如某个弹窗的瞬时开关或单次 hover 状态

### 3. 数据模型边界

- store 不得直接长期持有未经整理的原始接口脏数据
- 需要复用的接口响应应先转换为稳定的前端 ViewModel 再入 store
- store 中的字段命名必须面向业务语义，不得机械照搬后端命名

### 4. 持久化与副作用

- 持久化状态必须明确白名单，禁止默认全量持久化
- 与本地存储、路由、埋点、广播通道相关的副作用必须集中管理
- store 之间的依赖必须克制，避免形成循环引用和初始化顺序隐患

### 5. 错误与并发

- 异步 action 必须处理失败、取消、重复触发和竞态覆盖
- 列表筛选、分页、搜索、切换标签等场景必须明确最新请求优先策略
- 禁止在多个页面各自绕过 store 更新同一份核心全局状态

## 推荐实践

- 只把跨组件、跨页面、跨路由需要共享的稳定状态放进 Pinia
- 为会话、权限、筛选器、布局等状态设计清晰的 store API
- 将接口请求和数据转换组合为 store action，降低页面编排复杂度
- 对大型模块拆分多个聚焦 store，避免单个 store 膨胀

## 反例/禁用项

- 把所有表单项、弹窗开关、hover 标记都放进全局 store
- 在 store 里直接保存大块原始接口响应，页面各自再二次加工
- 多个 store 相互直接写状态，导致链式副作用难以排查
- 使用持久化插件时不加筛选，把敏感或无意义状态一并落盘

## 检查清单

- [ ] 全局状态、模块状态、组件状态边界清晰
- [ ] `state/getter/action` 职责分明
- [ ] store 不承载原始脏数据或纯视图临时状态
- [ ] 持久化策略、并发策略、失败策略明确
- [ ] store 依赖关系简单可维护
