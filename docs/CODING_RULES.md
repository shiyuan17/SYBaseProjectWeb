# CODING_RULES.md — 通用编码规范

## 目标与适用范围

本文件定义跨模块、跨页面通用的前端编码与审查基线。

- 适用对象：所有开发者与 AI 智能体
- 适用范围：Vue 组件、TypeScript 模块、脚本、测试、文档示例代码
- 规范定位：作为 Vue 与 TypeScript 细则的上位规则

## 强制规则

### 1. 命名与表达

- 名称必须表达业务意图，不得使用无语义缩写
- 禁止拼音命名、序号命名、含糊命名，如 `doIt`、`temp2`、`dataList`
- 同一概念在全项目中保持统一命名
- 禁止魔法字符串路由名、权限码、状态码散落在页面中

### 2. 职责与复杂度

- 组件、组合式函数、store、服务模块必须保持单一职责
- 新增逻辑优先复用现有能力，不得重复造轮子
- 复杂分支必须拆到命名清晰的函数、组合式函数或映射对象中
- 大块条件分支、深层嵌套、复制粘贴代码必须重构
- 简单优先：只实现当前成功标准需要的代码，不为单次需求新增策略模式、注册表、插件层、配置项或“以后可能会用”的扩展点
- 不为不可能或未定义的场景堆砌兜底逻辑；确需防御时，必须能说明真实触发路径和验证方式
- 如果实现明显可以更短、更直白且不损害可读性，应先简化再交付

### 3. 注释与文档

- 注释只解释“为什么”和“约束”，不重复“代码在做什么”
- 对外组件、共享能力、复杂规则必须具备说明注释或文档
- 代码与注释冲突时，必须同步修正文档

### 4. 错误与边界处理

- 不得吞错误、不记录上下文、不提示用户可理解的结果
- 异步流程必须显式处理加载、失败、空数据、重复提交、超时等状态
- 页面不得直接把接口返回原样透传到视图层
- 调试日志、`debugger`、临时 TODO、未清理 mock 逻辑不得残留到交付代码

### 5. 可维护性与兼容性

- 公共能力修改必须评估现有页面和模块的兼容性
- 默认值、空态文案、回退逻辑必须稳定且可预期
- 源码、Markdown、JSON/YAML、脚本、配置等文本类文件统一使用 `UTF-8`，默认采用 `UTF-8（无 BOM）`
- 文本文件编码默认跟随仓库 `.editorconfig` 的 `charset=utf-8`，避免不同编辑器、终端和工具链自行猜测编码
- 禁止在仓库内混用 `GBK`、`ANSI`、`UTF-16` 等文本编码；若第三方接口、历史存量文件或特定导出格式必须使用其他编码，必须显式说明原因、边界和验证方式
- 涉及中文、特殊符号、文件名、导出文本或复制粘贴链路时，必须显式验证编码在编辑器、终端、浏览器和目标打开方式中的一致性
- 时间、时区、精度、文件名、编码、浏览器能力必须显式处理
- 路径、命令、环境变量示例统一使用 `pnpm` 与跨平台可执行方式

### 6. 测试与自检

- 新增功能、缺陷修复、重要重构必须补充验证
- 测试至少覆盖正常路径、边界条件、失败分支
- 缺陷修复应优先写出能复现问题的测试或最小复现步骤，再修复并证明回归通过
- 新增校验必须覆盖无效输入；新增错误处理必须覆盖失败分支；重构必须说明行为保持不变的验证命令或对比方式
- 提交前必须完成自检，确认无明显重复、无死代码、无调试残留

标准验证命令（统一使用 `pnpm`，按改动范围选择最小有效集）：

| 场景 | 命令 | 说明 |
| --- | --- | --- |
| 代码规范 | `pnpm lint` | ESLint / oxlint 等统一校验 |
| 类型检查 | `pnpm check:type` | `vue-tsc` 类型检查（等价 `turbo run typecheck`） |
| 拼写检查 | `pnpm check:cspell` | cspell 校验（扫描 `**/*.{ts,tsx,vue}` 与 `**/README.md`，不覆盖其他 `*.md`） |
| 治理校验 | `pnpm run check:governance` | 校验治理文档关键约束（当前覆盖决策 ID、文档索引、架构快照、`PROJECT_STATE.md` 结构） |
| 综合静态检查 | `pnpm check` | 循环依赖 + 依赖 + 类型 + 拼写 + 治理一次性执行 |
| 单元测试 | `pnpm test:unit` | Vitest（`--dom`），逻辑/工具/组件 |
| 端到端测试 | `pnpm test:e2e` | Playwright，关键链路回归 |
| 构建 | `pnpm build` | 产物构建，发布前必过 |

- `pnpm check:cspell` 的扫描范围见上表（不覆盖 `docs/` 规范与根目录记忆文件等 `*.md`，改这些文件时需自行核对拼写）。改逻辑/组件时至少跑 `pnpm lint` + `pnpm check:type` + 相关 `pnpm test:unit`
- 改 `AGENTS.md`、`docs/` 顶层规范、记忆文件或治理脚本时，至少补跑 `pnpm run check:governance`
- 交付时在「验证结果」中写明实际执行的命令与结论，未执行项必须标注为未验证

## 推荐实践

- 优先通过类型、枚举、常量和映射表达规则，而不是靠注释约束
- 用统一错误对象、日志字段和提示策略提升问题定位效率
- 将页面数据转换、权限判断、格式化逻辑从模板中抽离
- 统一使用支持 `UTF-8` 的编辑器与终端设置，减少中文内容在协作和脚本输出中的误解码
- 对共享组件和基础能力补充最小使用示例，降低误用成本

## 反例/禁用项

- 复制一个页面改几个字段就当新功能
- 在模板中塞入大段三元表达式和复杂链式判断
- 请求失败后静默吞掉，只留空白页面
- 在多个页面各自手写相同的错误处理、分页协议或图表销毁逻辑
- 为了赶进度留下大量魔法值、临时样式和调试输出
- 为单一场景新增抽象工厂、策略注册、全局配置或未被使用的扩展点
- 修复一个 bug 时顺手重排无关代码、改写注释或改变旧行为

## 检查清单

- [ ] 命名清晰且术语统一
- [ ] 职责边界清晰，无明显重复逻辑
- [ ] 没有超出当前成功标准的 speculative abstraction、配置项或功能分支
- [ ] 复杂规则有注释或文档说明
- [ ] 错误、边界、空态处理完整
- [ ] 文本类文件编码、中文内容与导出文本链路已按 `UTF-8` 规则检查
- [ ] 兼容性影响已评估
- [ ] 测试和自检已完成，并能对应成功标准
