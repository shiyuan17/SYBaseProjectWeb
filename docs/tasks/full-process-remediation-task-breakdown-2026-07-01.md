# 全流程整改计划任务拆分（2026-07-01）

## Background

来自 `docs/reviews/full-process-remediation-plan-2026-07-01.html`，用于把三阶段整改计划收敛为可独立执行的本地任务项。

任务元数据以 `backlog.json` 为准，单任务执行语义以 `docs/tasks/T-00x-*.md` 为准。本文件只负责阶段映射、依赖关系和建议执行顺序，不重复审计结论。

## 拆分原则

- 第一阶段优先补主流程真实闭环和高风险写权限边界。
- 第二阶段补统一 tracking、统一审计、审批语义和并发控制。
- 第三阶段收敛状态模型、重复接口、展示规范和幂等日志。
- 每个任务只承接一类主目标，避免把“建模、接口、前端、治理”混成单个大包。

## 建议执行顺序

### 第一阶段：主流程闭环补齐

1. `T-002` 冰冻流程后端真实闭环
2. `T-006` 技术任务池权限拆分与服务校验
3. `T-004` 技术医嘱工作站后端动作契约补齐
4. `T-003` 冰冻工作台前端接入真实接口
5. `T-005` 技术医嘱工作站前端动作接线

说明：

- `T-002`、`T-004`、`T-006` 可并行推进，但都属于高风险跨前后端任务。
- `T-003` 必须在 `T-002` 接口契约稳定后接线。
- `T-005` 依赖 `T-004` 的动作契约和 `T-006` 的权限边界，不建议提前开工。

### 第二阶段：统一追踪、统一审计、借阅审批与并发一致性

1. `T-007` 申请单统一流程事件接入 tracking
2. `T-008` 正式报告分发事件接入 tracking
3. `T-010` M5 材料借阅审批流建模与接口落地
4. `T-009` 敏感接口统一权限与审计治理
5. `T-011` 归档借阅与白片库存并发安全加固

说明：

- `T-007`、`T-008`、`T-010` 可作为第二阶段三条主线并行推进。
- `T-009` 最好在 `T-006` 权限拆分规则落稳后执行，避免重复返工。
- `T-011` 必须建立在 `T-010` 新借阅状态机和接口之上。

### 第三阶段：状态模型、接口收敛、展示一致性与幂等治理

1. `T-012` M2 标本状态与异常流转模型重构
2. `T-013` 技术登记接口收敛
3. `T-014` 主状态与分发状态展示规范统一
4. `T-015` 批量动作幂等与操作日志标准化

说明：

- `T-012` 与 `T-013` 可以并行，但都属于“会影响后续展示与契约解释”的基础治理任务。
- `T-014` 需要等待报告分发、借阅审批、标本状态和技术登记主契约全部稳定。
- `T-015` 需要建立在报告分发事件和借阅并发控制完成后再统一收口。

## 任务分组说明

### A 组：立即进入执行队列

- `T-002`
- `T-004`
- `T-006`

适用原因：

- 都直接对应第一阶段“主流程闭环补齐”。
- 都会影响前端是否还能继续停留在 mock / 占位状态。
- 都属于后端支撑不充分或权限边界不成立的高优先级问题。

### B 组：跟随真实契约接线

- `T-003`
- `T-005`

适用原因：

- 依赖后端真实契约和权限边界落稳。
- 前端接线过早会导致 mock、字段映射和错误提示反复返工。

### C 组：治理与一致性补齐

- `T-007`
- `T-008`
- `T-009`
- `T-010`
- `T-011`
- `T-012`
- `T-013`
- `T-014`
- `T-015`

适用原因：

- 这批任务主要解决 tracking、审计、审批、并发、状态表达和幂等问题。
- 它们不一定阻塞第一阶段上线验证，但决定系统是否“可追、可审、可控、可维护”。
