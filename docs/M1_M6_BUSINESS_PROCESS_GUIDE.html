01. M1-M6 全局总览

全局闭环以“管理员准备 -> 标本送检 -> 技术生产 -> 医生诊断 -> 归档运营 -> 集成统计支撑”为主线组织。M1 负责底座与前置，M2-M5 负责常规病理主闭环，M6 负责医院外围系统对接、统计报表与历史数据支撑。

M1-M4：前端主能力已落地 M4 医嘱执行 / M6：后端能力明确 M5-M6 前端开放程度仍在推进
M1 管理底座

负责账号、角色、基础字典、系统配置和编号规则，保证后续岗位可用。

M2-M5 主闭环

覆盖从申请单创建到报告发布、再到材料归档与借阅回库的核心业务闭环。

M6 外围支撑

覆盖集成任务追踪、收费回写与对账、历史报告导入/查询、统计报表生成。

主闭环对象

申请单、标本、蜡块、包埋盒、玻片、诊断任务、病理报告、归档记录。

补充闭环对象

返工单、修订单、会诊单、病理医嘱、借阅单、集成任务、收费记录、历史报告、统计报表。

02. M1 系统管理

M1 作为全系统前置管理底座，负责账号、角色、基础字典、编号规则和系统配置， 其目标不是完成病例业务本身，而是保障 M2-M6 业务岗位能正确进入系统、看到对应菜单并使用统一规则。

系统用户：已落地 角色授权：已落地 基础字典/编号/配置：已落地
关键岗位：病理管理员 关键对象：用户、角色、权限、字典项、编号规则 目标：给 M2-M6 提供统一入口和规则底座
步骤 执行岗位 操作动作 输入/前置 输出/结果 异常点
1. 创建用户 病理管理员 在系统用户页新增 M2-M6 所需岗位账号，设置启停状态与基础信息。 岗位名单、账号策略、部门信息。 生成可登录账号。 账号编码重复、状态未生效、部门归属错误。
2. 分配角色 病理管理员 为账号绑定主角色和岗位角色，确认菜单与权限树。 后端角色种子、岗位权限定义。 账号具备对应模块入口和操作权限。 角色保存后菜单未刷新、岗位错配导致页面不可见。
3. 维护字典 病理管理员 维护部位字典、医嘱字典、收费字典、模板和取材规范等主数据。 业务编码标准、科室约定值。 业务页面具备稳定下拉项与默认模板。 字典缺项、命名不统一、前后端枚举口径不一致。
4. 配置编号与系统参数 病理管理员 设置申请单号、病理号、包埋盒号、玻片号等编号规则，并核对系统参数。 编号前缀、流水规则、系统配置项。 下游模块能按规则生成唯一编号并读取参数。 编号冲突、规则未发布、关键参数未回流页面。
5. 登录回归 对应业务岗位 重新登录系统，验证菜单、入口和基础动作是否可见。 前四步已完成。 M2-M6 账号可按岗位进入对应模块。 角色正确但菜单未可见、入口跳到旧路径、按钮缺权限。
关键单据/对象流转
  • 用户对象：账号状态、所属部门、主角色、岗位角色。
  • 角色对象：菜单树、权限树、查询/操作授权。
  • 基础字典：部位、医嘱、收费、模板、取材规范。
  • 规则对象：病理号、申请单号、玻片号、包埋盒号编号规则。
现状说明
系统用户：已落地 角色授权：已落地 字典/配置：已落地
  • 当前仓库已存在系统管理服务层、路由与测试。
  • M1 在业务上主要充当前置准备区，不直接承接病例流转。

03. M2 标本送检

M2 负责把临床申请、标本对象和送检交接链路结构化进入系统，是病理主闭环的入口。 主路径是“申请单创建 -> 标本登记 -> 固定核验 -> 转运交接 -> 标本签收 -> 流程追踪”， 补充路径包含第三方临床申请导入。

申请创建/登记/签收：已落地 流程追踪：已落地 第三方导入：已有接口与入口规划
关键岗位：申请创建岗、标本登记岗、固定核验岗、转运交接岗、签收岗、追踪岗 关键对象：申请单、标本、交接记录、签收记录、追踪事件
步骤 执行岗位 操作动作 输入/前置 输出/结果 异常点
1. 申请创建或导入 申请创建岗 / 临床导入岗 手工创建申请单，或从第三方系统导入临床申请。 患者信息、申请信息、来源系统数据。 形成申请单主记录，保留来源与外部单号。 重复导入、字段缺失、来源系统编码不一致。
2. 标本登记 标本登记岗 录入送检标本信息，建立申请单与标本对象关系。 申请单已存在，标本基础信息齐备。 生成标本记录、条码或对象编号。 标本数量不符、对象缺失、编号回写失败。
3. 固定核验 固定核验岗 核验固定状态、容器、数量和临床信息一致性。 标本已登记。 形成固定核验结果，异常标本进入提醒链。 未固定、固定不足、信息不匹配。
4. 转运交接 转运交接岗 完成送检过程中的交接登记，记录交接人和时间。 核验完成或允许先交接后补录。 形成交接事件和转运轨迹。 漏交接、重复交接、交接对象不完整。
5. 标本签收 标本签收岗 按申请单/标本完成签收，确认进入病理内部处理链。 交接记录可见。 标本进入后续 M3 任务生成链。 部分签收、漏签收、对象状态未推进。
6. 流程追踪 流程追踪岗 按申请单或病例查看送检、签收、异常提醒和后续流转情况。 事件链已累计。 可查询完整 M2 流程轨迹。 事件漏写入、查询口径不统一、异常摘要缺失。
关键单据/对象流转
  • 申请单：可来自手工创建或第三方临床导入。
  • 标本对象：与申请单关联，进入核验、交接、签收和追踪链。
  • 交接/签收记录：构成 M2 事件链，并作为 M3 上游前置。
  • 追踪摘要：承载状态、时间点、异常提醒和后续模块入口线索。
现状说明
主送检链路：已落地 第三方导入：已有能力与入口规划
  • M2 当前已形成独立工作流模块与统一入口。
  • 第三方临床导入在文档与服务层规划中占位，是本模块的重要补充路径。

04. M3 技术组生产线

M3 承担技术生产主线，覆盖待办任务池、取材、脱水、包埋、切片、染色、技术追踪和返工链路。 核心目标是把 M2 签收进入的病例转化为可供医生诊断的玻片和完整技术过程记录。

任务池/取材/脱水/包埋/切片/染色：已落地 技术追踪/返工：已落地 质控留痕：后端与追踪链已明确
关键岗位：任务池、取材岗、脱水岗、包埋岗、切片岗、染色岗、返工岗、技术追踪岗 关键对象：技术任务、蜡块、包埋盒、玻片、返工单、质控记录
步骤 执行岗位 操作动作 输入/前置 输出/结果 异常点
1. 任务池接单 技术任务池/各工位岗位 按任务类型、状态、病理号等查询待办，进入对应工作站。 M2 已签收并生成技术任务。 定位到可执行技术任务。 任务池未投递、工位权限不足、超时任务不可见。
2. 取材 取材岗 开始并完成取材，生成蜡块/取材结果与后续脱水任务。 待取材任务、病例上下文。 形成取材记录和下游任务。 对象数量不符、取材描述缺失、下游任务未生成。
3. 脱水与包埋 脱水岗 / 包埋岗 完成批次脱水、包埋和包埋盒信息回写。 蜡块对象、脱水批次、设备信息。 形成包埋盒与可切片对象。 批次状态错误、设备信息缺失、包埋盒号异常。
4. 切片与染色 切片岗 / 染色岗 完成切片数量、厚度和染色情况登记，产出玻片。 包埋盒、切片参数、染色参数。 形成可供诊断使用的玻片对象。 玻片数量异常、质量问题未记录、染色状态未更新。
5. 技术追踪 技术追踪岗 按病例查看技术任务链、标本链、蜡块链、玻片链和事件链。 病例已进入 M3 处理。 形成可追溯的技术履历。 对象链断裂、质控历史缺失、事件回写不完整。
6. 返工与质控留痕 返工岗 / 质控岗位 创建返工单、执行返工动作，并把质控评价写入追踪链。 可返工对象、质控规则。 返工单、返工结果和质控记录回流病例轨迹。 返工对象层级不匹配、返工后未回推主链、质控结果丢失。
关键单据/对象流转
  • 任务对象：待办任务按类型分发到不同工位。
  • 物理对象：标本 -> 蜡块 -> 包埋盒 -> 玻片。
  • 返工对象：可按病例、标本、蜡块、包埋盒、玻片发起返工。
  • 追踪摘要：技术事件、质控评价和返工记录都进入统一追踪链。
现状说明
主生产线：已落地 返工与追踪：已落地 质控扩展：已有结构与记录口径
  • M3 现有文档已明确复用 M2 接收作为上游前置，不重复建设登记接收工作站。
  • 返工作为补充链路单独记录，但不阻塞主玻片出片链先走通。

05. M4 医生诊断

M4 覆盖诊断分配、诊断任务签收与开始、工作台诊断、报告全生命周期、报告追踪、修订、会诊和病理医嘱执行。 主目标是把 M3 生产完成的病例推进到报告签发发布，并形成可审计的修订和会诊链。

分配/工作台/报告/追踪/修订/会诊：已落地 病理医嘱执行：已有后端能力 独立医嘱执行页：规划补齐
关键岗位：诊断分配岗、诊断岗、复核岗、签发岗、报告追踪岗、医嘱执行岗 关键对象：诊断任务、病理报告、报告版本、修订单、会诊单、病理医嘱
步骤 执行岗位 操作动作 输入/前置 输出/结果 异常点
1. 诊断分配 诊断分配岗 查询待诊断任务并分派给目标诊断医生与后续审核人。 M3 染色完成并生成诊断任务。 任务进入目标医生待办。 任务漏分配、分派目标错误、角色权限不匹配。
2. 签收与开始 诊断岗 签收诊断任务，进入工作台并开始诊断。 待诊断任务可见。 任务状态进入处理中。 重复签收、状态未切换、工作台上下文不完整。
3. 工作台诊断与报告创建 诊断岗 查看标本/蜡块/玻片上下文，创建并编辑病理报告。 病例上下文、技术追踪、诊断模板。 形成报告草稿与诊断结论。 上下文缺失、报告草稿丢失、病理号映射错误。
4. 提交、复核、签发、发布 诊断岗 / 复核岗 / 签发岗 按报告生命周期完成提交、复核、签发和正式发布。 报告草稿已完成。 形成正式发布报告，并可触发收费/归档等下游链路。 跨角色误用、驳回后状态未回退、发布后摘要未同步。
5. 报告追踪 报告追踪岗 查询报告版本链、任务链、事件链、修订链、会诊链和医嘱链。 病例已进入 M4。 形成完整诊断审计轨迹。 版本链断裂、事件不回显、摘要不同步。
6. 修订、会诊、医嘱执行 诊断岗 / 复核岗 / 签发岗 / 医嘱执行岗 已发布报告可发起修订；复杂病例可发起会诊；病理医嘱可查询、签收、完成或取消。 已发布报告、会诊需求或医嘱需求。 形成修订链、会诊链和医嘱事件链。 非主持人完成会诊、医嘱状态无法回写、独立执行页缺失导致入口不统一。
关键单据/对象流转
  • 诊断任务：由 M3 玻片完成后生成并进入分配链。
  • 病理报告：经历草稿、提交、复核、签发、发布，并维护版本链。
  • 修订/会诊：作为发布后的补充闭环，与工作台和追踪页互相回流。
  • 病理医嘱：与诊断上下文关联，可查询待执行、签收、完成和取消。
现状说明
诊断主闭环：已落地 修订/会诊：已落地 病理医嘱执行：已有后端能力 独立前端入口：规划补齐
  • 当前前端工作台和追踪页已显示医嘱摘要，但没有独立医嘱执行页面入口。
  • 本方案按理想全量业务纳入医嘱执行链，并明确标记其现状边界。

06. M5 归档与运营支撑

M5 负责病例材料的归档、借阅与回库，以及试剂、设备两类科室运营台账能力。 业务上承接 M4 报告发布后的归档动作，同时把档案位置和借阅状态回流给后续查询场景。

归档/试剂/设备接口与权限：已有后端能力 前端独立模块：规划补齐 拍照/扫码/打印：规划补齐
关键岗位：归档管理员、试剂管理员、设备管理员、借阅经办人 关键对象：归档柜、库位、归档记录、借阅单、试剂库存、设备档案、维护记录
步骤 执行岗位 操作动作 输入/前置 输出/结果 异常点
1. 归档柜与库位维护 归档管理员 创建/更新归档柜，查询可用库位，准备归档空间。 档案柜基础信息、容量规则。 形成可用归档空间。 容量不足、库位状态错误、柜编码重复。
2. 申请单/包埋盒/玻片归档 归档管理员 对已完成病例的申请单、包埋盒或玻片执行归档，绑定位置。 M4 报告发布、对象可归档。 形成归档记录与位置映射。 重复占位、对象状态不允许归档、位置回写失败。
3. 借阅与归还 借阅经办人 对在库材料发起借阅、登记借出，再执行归还回库。 归档位置已存在，材料在库。 形成借阅单、归还记录和最新在库状态。 重复借阅、重复归还、借阅状态未同步查询侧。
4. 试剂台账与库存预警 试剂管理员 维护试剂主数据、库存批次、近效期与低库存预警。 试剂基础信息、批次信息、阈值规则。 形成试剂台账与库存告警。 库存为负、批次过期、阈值不合理。
5. 设备台账与维护 设备管理员 维护设备档案、登记维护保养记录和设备预警。 设备编号、分类、维护计划。 形成设备状态、下一次维护时间和预警结果。 维护时间不合法、设备状态错误、预警未刷新。
6. 查询回流 M4/M6 查询侧 在医生工作台、报告追踪或历史场景中查看归档位置、借阅状态和物资支撑信息。 前述链路已形成记录。 下游模块获得可追溯的归档和运营支撑信息。 位置不回流、借阅状态滞后、跨模块摘要不一致。
关键单据/对象流转
  • 归档对象：申请单、包埋盒、玻片都可绑定归档位置。
  • 借阅对象：在库材料形成借阅单和归还记录。
  • 试剂对象：主数据 + 库存批次 + 预警。
  • 设备对象：设备档案 + 维护日志 + 到期预警。
现状说明
权限/菜单/接口：已有后端能力 前端模块：规划补齐 拍照/扫码/打印：规划补齐
  • M5 已有较完整后端菜单、权限和接口矩阵，但前端独立模块仍处于实现规划阶段。
  • 真实设备接入不在当前落地范围内，需要后续兼容性专项验证。

07. M6 集成 / 收费 / 历史 / 统计

M6 负责把病理主闭环与医院外围系统连接起来，围绕集成任务、收费回写与对账、历史报告导入/查询、 统计指标与报表生成构建外部支撑能力。它不是单独的病例生产链，而是贯穿 M2-M5 的外围能力层。

控制器/权限/测试基线：已有后端能力 临床导入/收费/历史/统计：已有代码与接口 前端统一门户：规划补齐

08. 其他流程专题

除模块主流程外，以下八类专题链路对业务培训和联调验收也很关键。 它们要么是主闭环的重要补充，要么是上线前必须明确的异常/支撑路径。

M3 返工 / M4 修订 / M4 会诊：已具主链支撑 M4 医嘱 / M6 支撑链：后端能力明确 M5 借阅运营门户：规划补齐
1. 常规病理主闭环

M1 -> M2 -> M3 -> M4 -> M5。管理员准备岗位与规则后，从申请单创建到标本送检、技术生产、诊断发布，再到归档回库形成常规主链。

  • 前置：账号、角色、字典、编号规则已就绪。
  • 结果：报告发布且材料归档可追溯。
2. 临床导入补充链路

第三方系统申请单导入后，直接汇合到 M2 登记链。重点关注来源系统、外部单号、幂等和失败补录机制。

  • 前置：集成任务可追踪。
  • 结果：导入申请单进入 M2 正常送检流程。
3. M3 返工链路

病例、标本、蜡块、包埋盒或玻片任一层级出现问题时，可发起返工单并回退到对应技术环节。

  • 重点：返工对象层级、返工原因、返工后状态回写。
  • 结果：返工事件进入技术追踪页。
4. M4 修订链路

报告已发布后，如需更正或补充，可由诊断岗发起修订申请，经审批后形成新版本链。

  • 重点：审批权限、版本链、旧版本可追溯。
  • 结果：工作台和追踪页同步显示修订摘要。
5. M4 会诊链路

复杂病例由诊断岗发起科内会诊，参与人提交会诊意见，主持人完成会诊并回流到摘要和追踪。

  • 重点：主持人权限、意见回显、完成动作审计。
  • 结果：会诊过程可追溯并影响最终诊断判断。
6. M4 医嘱执行链路

诊断过程中可创建病理医嘱，执行岗负责查询待执行医嘱、签收、完成或取消，形成医嘱状态链。

  • 重点：待执行查询、签收、完成、取消、追踪摘要。
  • 结果：医嘱事件回流工作台和报告追踪页。
7. M5 借阅归还链路

归档材料可因会诊、复查、教学等原因借出，再通过归还动作回到在库状态，并保留借阅历史。

  • 重点：在库校验、重复借阅/重复归还控制、位置回写。
  • 结果：医生查询侧能感知当前材料是否外借。
8. M6 收费/历史/统计支撑链路

报告发布或运营分析场景下，收费回执与对账、历史报告检索、统计指标报表都属于重要支撑链，虽不直接参与主生产，却影响上线完整性。

  • 重点：失败重试、历史版本、报表口径、任务可观测性。
  • 结果：外围系统链路可监控、可回放、可追责。
