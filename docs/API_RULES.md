# API_RULES.md — 接口与数据访问规范

## 目标与适用范围

本文件定义 `Axios` 请求层、接口模型、错误处理与数据转换规则。

- 适用对象：前端开发者、联调人员、AI 助手
- 适用范围：HTTP 请求、文件上传下载、分页列表、错误提示、接口类型定义
- 规范定位：统一“怎么调接口、怎么处理错误、怎么进入页面模型”

## 强制规则

### 1. Axios 实例与分层

- 项目必须通过统一 Axios 实例发起请求
- 请求配置、拦截器、超时、基础 URL、鉴权头等能力集中管理
- 业务页面不得直接在组件中临时创建 Axios 实例

### 2. 请求与响应模型

- 请求参数与响应结果必须有明确 TypeScript 类型
- DTO、响应对象、ViewModel 应分层定义，不得混用
- 页面与组件不得长期依赖后端原始结构，应通过 mapper 或 formatter 转成前端稳定模型

### 3. 错误处理

- 接口错误必须区分网络错误、超时、未认证、无权限、业务错误、服务异常
- 全局拦截器只负责通用策略，例如鉴权失效、基础提示、日志记录
- 页面级错误提示必须保留业务语境，不能全部依赖全局兜底提示

### 4. 分页、筛选与取消请求

- 分页参数统一使用 `page`、`size`
- 排序参数统一使用 `sort`
- 列表筛选参数命名必须稳定、清晰
- 可重复触发的查询必须支持取消请求、忽略过时结果或其他并发控制策略

### 5. 上传、下载与导出

- 文件上传必须校验类型、大小、失败提示和重试策略
- 下载与导出必须显式处理文件名、编码、浏览器兼容和失败兜底
- 长耗时导出必须提供用户可理解的状态反馈

### 6. 超时、重试与可观测性

- 接口超时必须统一配置并可按场景覆盖
- 重试只适用于明确幂等或可安全重试的场景
- 关键请求失败必须留下可排查信息，但不得泄露敏感数据

## 推荐实践

- 将接口定义按业务域放在模块内 `api/`
- 对列表查询、详情查询、表单提交建立统一返回模型和转换函数
- 用专门的 mapper 模块处理后端字段到前端字段的转换
- 对用户可修复错误与系统异常采用不同提示策略

## 反例/禁用项

- 在多个页面里重复手写 `baseURL`、Token 拼接和错误处理
- 组件直接依赖后端原始字段并在模板中散落格式转换
- 请求失败只弹“操作失败”而不区分错误类型
- 为了图省事对所有请求默认自动重试

## 检查清单

- [ ] 使用统一 Axios 实例和拦截器
- [ ] 请求参数、响应模型、ViewModel 已分层定义
- [ ] 错误处理区分通用策略和页面业务语境
- [ ] 分页、排序、筛选与并发控制规则统一
- [ ] 上传、下载、导出兼容性与反馈策略完整
