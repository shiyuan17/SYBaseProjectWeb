# 病理数据驾驶舱局部交互与布局优化设计

## 背景

当前病理数据驾驶舱已经具备完整的三列大屏结构，但有三个明显的可用性问题：

- 指标说明以整条横幅直接展示，占据了首屏有效空间，且信息密度过高
- `签发报告修改率` 一次性铺满全部月份记录，纵向占用较大，不利于首屏浏览
- `上月工作量` 当前采用轨道气泡式布局，视觉装饰较强，但指标之间的对比和扫描效率偏弱

这三个问题都发生在现有页面内部，不涉及统计口径、权限、接口契约或全局主题，只需要在当前 dashboard 模块内做局部 UI 交互增强。

## 目标

在不改变数据来源和页面整体信息架构的前提下，完成以下优化：

- 将指标说明收敛为一个小图标入口，不再长期占用大面积版面
- `签发报告修改率` 只展示固定可视区域，其余记录自动循环滚动
- 鼠标移入滚动区域时暂停，移出后继续滚动
- 将 `上月工作量` 调整为更清晰、可快速扫读的指标布局

## 非目标

- 不修改任何指标统计口径
- 不新增或调整后端接口字段
- 不修改三列主框架、页面路由或首页跳转行为
- 不将本次局部交互抽象为跨模块共享组件
- 不顺手改造其它面板的样式语言

## 现状问题

### 1. 指标说明横幅过重

当前位置：

- `apps/web-ele/src/modules/dashboard/components/pathology-screen/PathologyDashboardHeader.vue`

当前 `partialNotes` 直接渲染为整块横幅：

- 文案非常长时，会大幅压缩下方内容的首屏空间
- 说明内容是“按需查看”的辅助信息，但当前默认始终展开
- 大屏主标题与说明横幅争抢注意力，降低视觉重心

### 2. 签发报告修改率列表过长

当前位置：

- `apps/web-ele/src/modules/dashboard/components/pathology-screen/PathologyDashboardLeftColumn.vue`

当前按月份逐条完整渲染：

- 所有月份同时出现，导致左列高度快速增长
- 首屏只能看到列表的一部分，且没有任何浏览节奏控制
- 用户若想停留查看某一项，当前没有暂停机制

### 3. 上月工作量区域识别效率一般

当前位置：

- `apps/web-ele/src/modules/dashboard/components/pathology-screen/PathologyDashboardCenterColumn.vue`
- `apps/web-ele/src/modules/dashboard/styles/pathology-dashboard.css`

当前中心区 `上月工作量` 使用轨道、圆形气泡和中心装饰：

- 更偏展示感，而非阅读效率
- 数值、标签和层级的对齐关系较弱
- 在数据全部较小或相近时，视觉差异不足，信息重点不突出

## 设计决策

采用“保留现有三列结构，仅增强局部交互和布局”的方案：

1. 指标说明改为头部小图标按钮，点击后在原地弹出浮层
2. `签发报告修改率` 改为固定高度的纵向自动循环滚动列表
3. `上月工作量` 改为更规整的 2x2 指标卡片布局
4. 不引入新的共享层抽象，所有实现收敛在 dashboard 模块内

这样可以在最小 diff 下解决当前页面的三处体验问题，同时避免把一次局部大屏优化扩大成跨模块重构。

## 详细方案

### 1. 指标说明改为图标 + 原地浮层

影响位置：

- `PathologyDashboardHeader.vue`
- `pathology-dashboard.css`

调整内容：

- 当 `partialNotes.length > 0` 时，不再渲染整条 `pathology-warning`
- 在头部右侧增加一个小型说明按钮，默认展示为 `i` 信息图标
- 点击按钮后，在按钮附近原地展开浮层，展示：
  - 标题：`部分指标暂未完全就绪`
  - 内容：`partialNotes.join('；')`

交互要求：

- 再次点击图标可关闭浮层
- 点击浮层外区域可关闭浮层
- 按 `Esc` 可关闭浮层
- 浮层出现时不推动页面布局，不占据固定版面

可访问性要求：

- 使用原生 `button`
- 按钮补 `aria-label`
- 按钮维护 `aria-expanded`
- 浮层以可感知标题和正文组织，避免仅依赖颜色传达状态

### 2. 签发报告修改率改为可暂停循环滚动

影响位置：

- `PathologyDashboardLeftColumn.vue`
- `pathology-dashboard.css`

调整内容：

- 外层列表增加固定可视高度，只显示前几项内容
- 当条目数超过可视阈值时，启用自动纵向滚动
- 为实现无缝循环，渲染时复制一份列表内容接在后面
- 使用 CSS 动画驱动滚动，不引入额外计时器复杂度

交互要求：

- 鼠标移入滚动容器时暂停动画
- 鼠标移出后恢复动画
- 当条目数不足以滚动时：
  - 不复制列表
  - 不启动动画
  - 保持静态展示

视觉要求：

- 保留当前单条样式基调
- 让列表容器边界更明确，避免滚动时产生“内容断层感”
- 不新增夸张动效，滚动速度以可读为先

### 3. 上月工作量改为 2x2 指标卡布局

影响位置：

- `PathologyDashboardCenterColumn.vue`
- `pathology-dashboard.css`

调整内容：

- 中央 `center-stage` 区域不再使用核心圆心 + 分散气泡作为主要信息载体
- 保留一定的大屏风格装饰线，但信息主体改为四个规整卡片
- 四个卡片按 2x2 排列，继续展示：
  - 切片
  - 脱水
  - 包埋
  - 报告

每张卡片展示：

- 指标标签
- 指标值
- 状态样式

布局目标：

- 数值成为第一视觉焦点
- 标签与数值对齐关系清晰
- 在 4 个指标同时为 0 或数值接近时，仍然具备良好的扫读体验

视觉约束：

- 继续沿用当前大屏蓝色系、发光边框和深色背景
- 降低纯装饰性轨道图形比重
- 让 `上月工作量` 标题与卡片区形成一个完整模块

## 风险与约束

### 风险 1

头部浮层如果定位或层级处理不当，可能被大屏其它区域遮挡，或在窄宽度下溢出。

缓解：

- 让浮层相对按钮容器定位
- 明确 `z-index`
- 控制最大宽度与自动换行

### 风险 2

滚动列表若直接依赖 JavaScript 定时器，容易引入清理、卡顿和测试不稳定问题。

缓解：

- 优先采用 CSS 动画 + 内容复制
- 仅在“是否滚动”判断上使用轻量计算

### 风险 3

上月工作量重排后，若完全移除中心区装饰，可能让页面中轴显得过于平。

缓解：

- 保留少量底座/描边/背景光效
- 让信息卡承担主体，装饰只做辅助

## 验收标准

### 场景 1

当页面存在部分指标说明时：

- 首屏不再显示整条说明横幅
- 头部显示一个小型说明入口
- 点击后能看到完整说明内容

### 场景 2

当 `签发报告修改率` 条目较多时：

- 首屏只显示固定可视区内容
- 列表自动循环向上滚动
- 鼠标移入后滚动暂停
- 鼠标移出后继续滚动

### 场景 3

当查看 `上月工作量` 区域时：

- 四个指标以更规整的方式排列
- 数值与标签更容易扫读
- 不再以大面积分散圆形气泡作为主要信息结构
