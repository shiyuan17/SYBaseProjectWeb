M3-M4 冰冻专项

冰冻流程专项实施方案

目标形态采用“不新增一级业务域、在技术组与医生组内新增冰冻专项入口”。 通过显式 FrozenSession 与阶段任务模型，替代旧 WinForm 中大量依赖 isFrozen 参数分支的实现方式，先以 Web mock 流程跑通术中冰冻主链路，再平滑切换真实后端。

1. 当前实现结论

  • 旧系统冰冻不是单页闭环，而是预约、接收、取材、切片、快速报告、冰石对比、提醒看板和交接班的跨页组合。
  • 技术侧主要通过 FrmReceive(1)、FrmPlan(true)、FrmSectionV2(true) 等参数化复用页面承接。
  • 报告侧通过 FrmFrozenMetion、FrmRep、FrmIceFrozen 分别处理提醒、快速报告与冰石对比。
  • 问题不在功能缺失，而在规则散、状态隐式、长期维护成本高。

2. 新系统落点

  • 技术组新增 /technical-workflow/frozen，承接冰冻接收、取材、切片和提醒。
  • 医生组新增 /doctor-workflow/frozen-report，承接初步结果、电话回报、最终确认、冰石对比和剩余组织处理。
  • 临床送检侧仅保留“冰冻进度查看”和与常规流程的关联展示，不回收全部作业动作。
  • 前端先统一走模块内 mock，会话数据源固定为 mock_data/frozen-workflow/frozen-sessions.json。

3. 目标状态机

REQUESTED：术中申请已创建，等待冰冻接收。
RECEIVED：完成冰冻接收，进入取材。
GROSSING：完成取材，进入切片。
SLICING：完成切片，进入快速报告。
DIAGNOSING / REPORTED：完成术中初步结果与电话回报。
CONFIRMED：快速报告最终确认。
PARAFFIN_REVIEWED：完成冰石对比。
CLOSED：剩余组织处理与交接班闭环完成。

4. 页面与职责映射

工作域 入口 本轮承接 暂不纳入
技术组 /technical-workflow/frozen 接收、取材、切片、提醒看板、时间线 真实打印设备、值班消息推送
医生组 /doctor-workflow/frozen-report 初步结果、电话回报、最终确认、冰石对比、剩余组织 真实报告编辑器、正式签发联动
临床送检 既有 M2 追踪视图 进度查看、是否已追加常规、冰冻会话关联 完整冰冻技术作业

5. 实施分期

  1. 前端先落 mock 会话模型、技术组/医生组专项入口与最小状态机。
  2. 补真实后端接口时，保持前端入口不变，只替换服务层实现。
  3. 后端以 frozen_sessions 为主表扩展任务、提醒、冰石对比与剩余组织记录。
  4. 联调时优先验证“技术侧完成切片后，医生侧可直接接续快速报告”。

6. 验收重点

  • 技术组冰冻入口可完成接收 → 取材 → 切片，状态会同步推到医生侧。
  • 医生组冰冻入口可完成初步结果 → 电话回报 → 确认 → 冰石对比 → 剩余组织处理。
  • 同一冰冻会话在技术侧与医生侧展示的是同一条时间线，而不是两套独立假数据。
  • Chromium 下中文无乱码，入口路径、标题和时间线文本与页面实际一致。
