M2 Clinical Workflow Plan

临床送检流程完善与执行方案

本方案面向当前仓库的 M2 临床送检闭环，目标是在不拆分顶层 4 个主入口的前提下， 逐步补齐外部临床病理工作站分析文档中强调的 11 类能力，并形成可执行、可联调、可验收的前后端改造路径。

保留 4 个主入口 补齐 11 类工作站能力 前后端闭环方案 输出正式 HTML 交付页
4 当前前端主入口：申请与登记、固定与转运、病理接收、追踪与异常
11 目标工作站能力：登记、核对、记录、条码、时间、确认、出入库、综合信息、异常管理
4 阶段 方案交付、前端显式能力、后端补口联调、回归验收与 SOP 同步

1. 现状摘要与目标原则

当前实现

当前系统已经落地 M2 主链路：申请创建/导入 -> 标本登记 -> 固定 -> 转运 -> 接收 -> 追踪。 前端形态收敛为 4 个合并工作台，后端已有申请、登记、固定、转运、接收、追踪主链路接口与测试基线。

目标原则

对齐外部 clinical-pathology-workstation-analysis.html 的工作站模型， 但不照搬 WinForm 菜单拆分方式；本轮优先保留现有 4 个顶层入口，采用页内场景拆分、提示增强、聚合字段扩展来补齐能力。

本轮边界

默认不修改全局路由守卫、Axios 全局拦截器和顶层权限模型。若后续出现“异常修正必须独立授权”这类红区变更，再单独发起人工确认。

3. 目标工作站映射关系

工作台一：申请与登记

对应目标能力：申请查询、患者核对、登记录入、标签补打、离体时间采集、异常补录入口。 继续使用 /workflow/submission-registration，通过页内卡片、分栏和对话框拆分子场景。

工作台二：固定与转运

对应目标能力：标本核对、标本固定、条码绑定、标本确认、标本入库、转运/出库。 继续使用 /workflow/fixation-transport，从“双标签页”演进为“6 个子场景工作台”。

工作台三：病理接收

对应目标能力：转运单接收、条码直收、部分接收、拒收、退回、质控问题记录、未接收提醒、申请单补打印承接位。 继续使用 /workflow/pathology-receipt，重点强化异常闭环。

工作台四：追踪与异常

对应目标能力：标本综合信息、申请追踪、标本追踪、核对记录、异常修正说明与回查。 继续使用 /workflow/tracking-exception，承担综合台账和例外处理入口说明。

兼容入口策略

保留旧入口 /workflow/clinical-register、/workflow/fixation-verify、 /workflow/transport-handover、/workflow/specimen-receipt、 /workflow/tracking-query 作为兼容跳转，不改变现有对外路径语义。

4. 前端改造清单

申请与登记

  • 增加申请单号扫码/输入检索头部区，统一“查询、重复申请提醒、患者核对状态、报告状态/检查状态”展示。
  • 把登记页内显式拆成“登记录入”“标签补打”“条码处理”“异常补录”子场景。
  • 增加离体时间采集入口，明确与送检时间、固定时间的展示顺序。

固定与转运

  • 保留顶层合并页，但从 2 个 tab 扩展到 6 个子场景：标本核对、标本固定、条码绑定、标本确认、标本入库、转运/出库。
  • 核对场景只保留“开始核对 / 完成核对”，固定场景只保留“开始固定 / 完成固定”，不再混放动作。
  • 转运单创建前必须满足“已核对、已固定、已确认、已入库”，并增强打印、交接、异常批次标记和“待出库/已出库”语义。

病理接收

  • 保留按转运单接收和按条码直收。
  • 强化部分接收、拒收、退回、质控问题、容器数量异常的交互说明与结果回显。
  • 增加未接收数量、异常批次提醒和申请单补打印承接位。

追踪与异常

  • 申请追踪页展示全流程节点、当前节点、异常标记、时间线和接收摘要。
  • 标本追踪页展示条码、状态、固定/接收结果、最近事件和异常原因。
  • 增加核对记录视图和异常修正入口说明，异常修正仍收敛在现有管理能力内。

6. 实施分期与风险

阶段一：方案页与差异基线

交付正式 HTML 方案页，确定目标工作站映射、差异矩阵、字段清单和验收范围。

阶段二：前端显式能力补齐

优先实现不依赖新接口的页内结构调整、核对/固定/确认/入库场景拆分、异常提示和信息聚合展示。

阶段三：后端补口与联调

补齐条码绑定、离体时间、核对记录、标本确认、接收异常摘要等接口和聚合字段，完成前后端联调。

阶段四：E2E 回归与 SOP 同步

完成主链路与异常链路回归，输出岗位 SOP、试点培训材料和联调记录。

风险一：工作站形态理解偏差

若把 WinForm 的 11 个菜单直接搬成前端一级菜单，会抬高培训和路由复杂度，不符合本轮“保留 4 个主入口”的约束。

风险二：后端字段未补齐

若只做前端界面拆分而不补聚合字段，会导致患者核对、接收摘要、核对记录等场景仍只能靠人工判断。

风险三：异常流程继续隐含

部分接收、拒收、退回、容器问题等若缺乏醒目提示和结果回显，现场仍会把异常标本硬推进到正常流程。

7. 验收标准与测试范围

单元测试

  • 新增场景切换、状态映射、异常标签、提醒计数、差异字段格式化。
  • 核对记录、接收摘要、批次异常标识等前端转换逻辑。

前端集成测试

  • 申请查询与登记
  • 标签补打
  • 核对开始 / 完成
  • 固定开始 / 完成
  • 标本确认 / 入库
  • 转运建单 / 打印 / 交接
  • 接收 / 部分接收 / 拒收 / 退回
  • 追踪页异常展示

后端集成测试

  • 条码绑定
  • 离体时间设置
  • 标本确认
  • 标本入库
  • 部分接收后的申请状态与轨迹
  • 异常质控问题写入与查询聚合

E2E 与文档验收

  • 正常主链路：申请 -> 登记 -> 核对 -> 固定 -> 确认 -> 入库 -> 转运 -> 接收 -> 追踪
  • 异常主链路：拒收/部分接收 -> 异常标记 -> 追踪回查 -> 异常修正
  • HTML 文档在 Chromium 下正常打开、中文无乱码、章节完整可读。
验收标准 1： 四个主入口不变，但 11 类目标能力在页面结构或接口设计中均能找到明确承接位。
验收标准 2： 部分接收、拒收、退回、容器质量问题等异常分支具备可视化提示、结果回显和追踪落点。
验收标准 3： 申请追踪和标本追踪可回看患者核对、时间节点、当前节点、异常摘要和最近操作上下文。

8. 假设与默认

默认假设

  • 以“保留 4 个合并工作台、补齐 11 类能力”为最终形态。
  • 本轮不修改全局权限模型、路由守卫、Axios 全局拦截器。
  • 标本入库 本轮纳入 /workflow/fixation-transport 页内独立场景，不新增顶层菜单。

配置分支处理原则

  • 患者核对、接收模式、申请单打印按钮等配置型能力，本轮先做前端可承接位和字段设计。
  • 不直接硬编码医院特定模式，避免把现场配置误写成统一业务标准。
  • 若院区存在明显差异，再在后续阶段通过配置驱动实现。
