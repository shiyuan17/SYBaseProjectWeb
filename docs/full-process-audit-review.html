01. 审查说明与口径

本次审查不是需求蓝图评审，而是现实落地审查。结论只基于当前代码可证明的事实，不基于产品愿景或历史计划推断。

事实时间点：2026-05-26 口径：真实落地
证据优先级

前端路由、页面、service、菜单映射、权限常量为一手证据；后端 controller、权限种子、集成测试为二手交叉验证；文档只作背景对照。

闭环判定档位

已闭环、部分闭环、后端具备但前端未闭环、仅文档存在、未落地 五档。

排除项

本次不把打印机、扫码枪、真实外部系统、真实收费回调链路的现场环境验收，等同为代码未闭环；但会单独标为目标环境待验。

审查边界
  • 前端仓库：apps/web-ele/src/router/routes/modules/*、apps/web-ele/src/modules/*/api/*、apps/web-ele/src/api/core/menu-mapper.ts、各模块入口页与视图。
  • 后端仓库：bl-center/src/main/java/.../controller/*、bl-center/src/main/resources/db/migration/*、代表性集成测试类。
  • 对照文档：docs/M4_M5_ARCHIVE_OPERATION_SUPPORT_IMPLEMENTATION_PLAN.md、docs/M1_M6_BUSINESS_PROCESS_GUIDE.html、docs/phase1_5/*。

    • 02. 总体结论与风险分级

    系统不是“未完成”，也不是“全量闭环”。更准确的判断是：常规病理主链已经跑通到归档，支撑链仍存在明确断点。

    总评：部分闭环
    主业务主链

    M1 -> M2 -> M3 -> M4 -> M5 代码证据完整，已形成从送检、技术生产、诊断到归档回填的闭环。

    M4 角色断层

    病理医嘱后端已具备创建、查询、接单、完成、取消能力，但前端没有独立工作台，执行角色无法闭环。

    M6 支撑链断层

    集成任务、收费管理、历史报告后端已落地并有集成测试，前端仍是 coming-soon，支持岗无法收口。

    文档治理偏差

    M5 在当前代码中已经实现前端主工作台，但旧实施计划仍将其表述为“未实现”，会误导验收与排期。

    High Risk 病理医嘱执行链对执行角色不可用

    医生可以产生医嘱摘要，后端可以查询与执行，但执行岗在前端没有入口。这不是体验问题，而是责任交接断点。

    High Risk M6 三条支撑链被占位页截断

    集成任务、收费管理、历史报告均已在后端形成控制器与测试，但前端没有可操作门面，导致角色权限存在而任务不可完成。

    Medium Risk 文档现状落后于代码现状

    如果仍以旧计划文档为验收依据，会把已交付能力误判为未交付，带来重复排期与错误风险说明。

    本系统的准确产品判断应当是：M1-M5 常规病理业务链已能闭环，M6 仍是后端先行、前端未跟上；M4 医嘱执行是当前最清晰的角色缺口。

    • 05. 关键断层、重复环节与缺失角色 Top Findings

    以下问题按严重度排序。结论优先，不做温和包装。

    Finding 1 / High M4 病理医嘱执行链断在前端角色落点

    后端已经把病理医嘱纳入正式菜单、权限和执行角色，前端却没有独立工作台。这意味着“创建医嘱”和“执行医嘱”被拆成了两个世界：医生侧能看到摘要，执行侧没有工作面。

    • 前端证据：doctor-workflow.ts 无医嘱路由；constants.ts 无 PERM_M4_MEDICAL_ORDER_*；DiagnosisWorkbenchView.vue 与 ReportTrackingView.vue 只做摘要展示。
    • 后端证据：PathologyMedicalOrderController.java 已提供 create / pending / accept / complete / cancel；V19__seed_m4_batch2_authorization.sql 已种下 ROLE_M4_MEDICAL_ORDER_EXECUTE 及权限。
    • 影响：角色交接不成立。系统可以生成医嘱，但不能保证执行岗在产品内完成自己的职责。
    Finding 2 / High M6 三条支撑链后端成熟、前端未交付

    integration、billing、history 在后端已有 controller、权限种子和集成测试，但前端路由仍统一指向 coming-soon。这不是“页面还没美化”，而是角色无法收口。

    • 前端证据：apps/web-ele/src/router/routes/modules/m6.ts 把 /m6/integration、/m6/billing、/m6/history 指向占位页；menu-mapper.ts 同步把后端菜单映射到占位页。
    • 后端证据：M6ManagementController.java 已暴露 integration、billing、historical-report、stat-report 相关接口；V27__seed_m6_authorization_and_indicator_defs.sql 已完整种下菜单与权限。
    • 测试证据：M6BillingIntegrationTest、M6HistoricalReportIntegrationTest、M6StatisticsIntegrationTest、M6AuthorizationIntegrationTest、M6ObservabilityIntegrationTest。
    • 影响：管理员、归档岗、质量岗在支持链上只能看到“能力存在”，不能实际完成操作。
    Finding 3 / Medium M5 已实现，但旧实施计划仍在误报“未实现”

    当前代码里 operation-support 已经包含路由、入口页、archive / reagents / equipment 三个工作台、service 与测试；旧文档仍把它描述为待实现，说明治理文档已落后于代码。

    • 前端证据：apps/web-ele/src/modules/operation-support/*、operation-support.ts、operation-support-service.test.ts、access.test.ts。
    • 后端证据：ArchiveWorkflowIntegrationTest、ArchiveRoleAuthorizationIntegrationTest、M5 controller 与权限种子均已存在。
    • 对照文档：docs/M4_M5_ARCHIVE_OPERATION_SUPPORT_IMPLEMENTATION_PLAN.md 仍把 M5 表述为“尚未实现”。
    • 影响：验收、排期、对外汇报极易出现重复说明和错误风险判断。
    Finding 4 / Low 未发现真正影响业务的重复环节

    当前代码中的“重复”主要是兼容别名、菜单映射和统一入口跳转，不构成业务重复录入或重复审批。把这些别名误判成流程重复，会得出错误结论。

    • 示例：/workflow/application-list、/workflow/clinical-register 最终回到同一登记页；这属于兼容路径，不是多套业务流程。
    • 示例：menu-mapper.ts 中大量 pathAliases 用于兼容后端菜单路径，不是重复页面实现。
    • 结论：当前更大的问题不是重复，而是角色断层与前后端交付不对齐。

    06. 关键路径可度量性评估

    可度量性统一按五项检查：是否有明确输入、是否有状态迁移、是否有责任人 / 操作人记录、是否有追踪 / 审计视图、是否有测试或观测证据。

    关键路径 输入 状态迁移 责任人 / 操作人 追踪 / 审计 测试 / 观测 得分 评价
    M2 申请到病理接收 有 有 有 有 有 5 / 5 路径合理，且具备前后端双侧验证。
    M3 技术生产到诊断任务生成 有 有 有 有 有 5 / 5 路径合理，返工与追踪已纳入度量对象。
    M4 诊断报告主链 有 有 有 有 有 5 / 5 报告生命周期清楚，版本链和追踪链完整。
    M5 归档 / 借阅 / 回填 有 有 有 有 有 5 / 5 已能反向回填到医生侧视图，是完整闭环的一部分。
    M4 病理医嘱执行支持链 有 有 有 有 有 5 / 5 后端可度量，但前台不可达；这是“可度量但不可交付”的典型断层。
    M6 集成 / 计费 / 历史支持链 有 有 有 有 有 5 / 5 后端具备完整度量基础，但前端缺工作面，导致角色无法利用这些度量能力完成任务。
    M6 统计分析 有 有 有 有 有 5 / 5 统计分析本身可用，问题不在统计页，而在 M6 其余三条支持链未交付。

    结论重点：本项目当前的主要风险不是“无法度量”，而是“部分路径虽然可度量，却没有交给对应角色可用的前台入口”。这会让系统在数据上看似完整，在业务上却无法闭环。

    07. 整改优先级建议

    整改顺序应当围绕真实断层，而不是围绕页面数量。优先补齐角色可操作性，再清理文档偏差。

    P0 补齐 M4 病理医嘱执行工作台

    新增独立路由、权限常量、service、菜单映射和页面工作台，至少覆盖 pending / accept / complete / cancel 主操作。否则医生链永远停在“可创建、不可执行”。

    P0 补齐 M6 integration / billing / history 前端

    如果短期不能交付，就必须从产品口径上明确降级并隐藏菜单；继续保留“有权限但落占位页”的状态不可接受。

    P1 同步修正文档基线

    将 M5 已实现、M6 部分已实现、M4 医嘱执行缺口等现状写回正式文档，避免后续验收和汇报继续沿用过期说法。

    P1 建立角色可达性清单

    每个后端种子角色都应有一张前端可达性清单，明确“入口页面、菜单映射、主操作、测试类”。这样可以提前发现“有角色、无页面”的断层。

    P2 继续保留兼容别名，但文档化其目的

    M2 的兼容路由和菜单别名不是问题，问题在于它们容易被外部评审误解成重复流程。应在规则文档中显式说明这是兼容策略。

    P2 把“主链闭环”和“支持链闭环”分开汇报

    后续对外表达应明确：常规病理主链已闭环到 M5；系统全量口径仍是部分闭环。混用这两个口径，会持续制造误判。

    附录说明

    这些点容易被误判，单独列明，避免报告被过度解读。

    1. 不计为业务重复的内容
    • /workflow/application-list、/workflow/clinical-register 等兼容路由最终导向统一页，不算重复业务环节。
    • menu-mapper.ts 里的 pathAliases 是后端菜单兼容策略，不算重复页面实现。
    • 入口页自动跳转到首个可用工位，是权限导航策略，不算流程多走一步。
    2. 未纳入代码闭环判断但需现场验证的内容
    • 真实打印机、扫码枪、设备 SDK、外部收费系统、真实历史导入数据源。
    • 这些属于目标环境验收，不应反向覆盖当前代码闭环判断，但必须在上线前补验。
    最终判断

    以当前代码事实为准，SYBaseProject 的常规病理业务主链已经从 M1 跑通到 M5；系统全量口径仍为部分闭环。 真正阻断全量闭环的，不是 M2-M5 主线，而是 M4 病理医嘱执行角色无前台落点 与 M6 integration / billing / history 三条支撑链前端未交付。
