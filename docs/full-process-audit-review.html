    • 05. 关键断层、重复环节与缺失角色 Top Findings

    以下问题按严重度排序。结论优先，不做温和包装。

    Finding 1 / High M4 病理医嘱执行链断在前端角色落点

    后端已经把病理医嘱纳入正式菜单、权限和执行角色，前端却没有独立工作台。这意味着“创建医嘱”和“执行医嘱”被拆成了两个世界：医生侧能看到摘要，执行侧没有工作面。

    • 前端证据：doctor-workflow.ts 无医嘱路由；constants.ts 无 PERM_M4_MEDICAL_ORDER_*；DiagnosisWorkbenchView.vue 与 ReportTrackingView.vue 只做摘要展示。
    • 后端证据：PathologyMedicalOrderController.java 已提供 create / pending / accept / complete / cancel；V19__seed_m4_batch2_authorization.sql 已种下 ROLE_M4_MEDICAL_ORDER_EXECUTE 及权限。
    • 影响：角色交接不成立。系统可以生成医嘱，但不能保证执行岗在产品内完成自己的职责。
    Finding 2 / Medium M6 已补齐，后续重点转为目标环境验收

    integration、billing、history 已经由真实页面承接，前端不再是占位状态。当前真正需要推进的是外围系统联调、历史数据源核验和现场角色菜单配置。

    • 前端证据：apps/web-ele/src/router/routes/modules/m6.ts、menu-mapper.ts、m6-management/* 已映射并实现四个真实入口页。
    • 后端证据：M6ManagementController.java 已暴露 integration、billing、historical-report、stat-report 相关接口；V27__seed_m6_authorization_and_indicator_defs.sql 已完整种下菜单与权限。
    • 测试证据：m6.test.ts、m6-management-service.test.ts、m6-management-views.test.ts 以及既有 M6*IntegrationTest。
    • 影响：若不做现场验收，风险会从“前端缺口”转成“联调与实施偏差”。
    Finding 3 / Medium M5 已实现，但旧实施计划仍在误报“未实现”

    当前代码里 operation-support 已经包含路由、入口页、archive / reagents / equipment 三个工作台、service 与测试；旧文档仍把它描述为待实现，说明治理文档已落后于代码。

    • 前端证据：apps/web-ele/src/modules/operation-support/*、operation-support.ts、operation-support-service.test.ts、access.test.ts。
    • 后端证据：ArchiveWorkflowIntegrationTest、ArchiveRoleAuthorizationIntegrationTest、M5 controller 与权限种子均已存在。
    • 对照文档：docs/M4_M5_ARCHIVE_OPERATION_SUPPORT_IMPLEMENTATION_PLAN.md 仍把 M5 表述为“尚未实现”。
    • 影响：验收、排期、对外汇报极易出现重复说明和错误风险判断。
    Finding 4 / Low 未发现真正影响业务的重复环节

    当前代码中的“重复”主要是兼容别名、菜单映射和统一入口跳转，不构成业务重复录入或重复审批。把这些别名误判成流程重复，会得出错误结论。

    • 示例：/workflow/application-list、/workflow/clinical-register 最终回到同一登记页；这属于兼容路径，不是多套业务流程。
    • 示例：menu-mapper.ts 中大量 pathAliases 用于兼容后端菜单路径，不是重复页面实现。
    • 结论：当前更大的问题不是重复，而是角色断层与前后端交付不对齐。

    06. 关键路径可度量性评估

    可度量性统一按五项检查：是否有明确输入、是否有状态迁移、是否有责任人 / 操作人记录、是否有追踪 / 审计视图、是否有测试或观测证据。

    07. 整改优先级建议

    整改顺序应当围绕真实断层，而不是围绕页面数量。优先补齐角色可操作性，再清理文档偏差。

    P1 冻结旧结论，统一对外口径

    把补齐前的“缺口版”结论标记为历史状态，后续统一以当前代码事实沟通，避免 M4 / M6 再被误报为未闭环。

    P1 执行现场验收清单

    围绕外部收费回执、历史导入真实数据源、CSV 编码 / 文件名、角色菜单可见性和工作面可达性开展目标环境核验，不再把这类问题留给后续口头补充。

    P1 同步修正文档基线

    将 M5 已实现、M4 医嘱执行已补齐、M6 四页真实可用等现状写回正式文档，避免后续验收和汇报继续沿用过期说法。

    P1 建立角色可达性清单

    每个后端种子角色都应有一张前端可达性清单，明确“入口页面、菜单映射、主操作、测试类”。这样可以提前发现“有角色、无页面”的断层。

    P2 继续保留兼容别名，但文档化其目的

    M2 的兼容路由和菜单别名不是问题，问题在于它们容易被外部评审误解成重复流程。应在规则文档中显式说明这是兼容策略。

    P2 把“主链闭环”和“支持链闭环”分开汇报

    后续对外表达应明确：本轮代码基线下 M1-M6 主能力已闭环；若谈现场上线状态，则需单列目标环境验收结果。不要再把“代码闭环”与“实施验收”混成一个口径。

    附录说明

    这些点容易被误判，单独列明，避免报告被过度解读。

    1. 不计为业务重复的内容
    • /workflow/application-list、/workflow/clinical-register 等兼容路由最终导向统一页，不算重复业务环节。
    • menu-mapper.ts 里的 pathAliases 是后端菜单兼容策略，不算重复页面实现。
    • 入口页自动跳转到首个可用工位，是权限导航策略，不算流程多走一步。
    2. 未纳入代码闭环判断但需现场验证的内容
    • 真实打印机、扫码枪、设备 SDK、外部收费系统、真实历史导入数据源。
    • 这些属于目标环境验收，不应反向覆盖当前代码闭环判断，但必须在上线前补验。
    最终判断

    以当前代码事实为准，SYBaseProject 的 M1-M6 当前前后端主能力已经闭环。 本轮已关闭的关键断层包括 M4 病理医嘱执行工作台 与 M6 integration / billing / history 三条真实前台链路；剩余工作集中在治理同步和目标环境验收。
