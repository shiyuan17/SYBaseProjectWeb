SYBaseProject 全流程整改实施计划

基于全流程审计报告形成的独立整改计划页面，仅承载分阶段实施计划，不复写审计问题清单。

计划日期：2026-07-01
来源报告：full-process-audit-2026-07-01.html
计划文件：full-process-remediation-plan-2026-07-01.html
目标定位：按阶段推进全流程整改、验收与排期

封面摘要

本计划的定位：将当前“分阶段调整实施计划”单独落为独立 HTML 文件，不并入现有审计报告，内容只保留计划本身，不附带审计摘要。

执行原则：先补主流程真实闭环，再补权限、审计、审批与一致性，最后收敛状态、接口和展示规范。页面内容完全以已确认计划为准，不新增新的业务结论，也不混入审计问题表。

总体策略

策略一：先补真闭环

优先消除“前端已开放但流程不成立”与“高风险写操作无正确边界”的问题，让核心链路先真实可用。

策略二：再补一致性

把统一追踪、统一审计、借阅审批与并发控制纳入第二阶段，避免系统处于“能跑但不可追、不可审”的状态。

策略三：最后做治理优化

在主流程稳定后，再收敛状态模型、重复接口、展示规范和幂等策略，降低后续维护成本。

本计划默认采用“直接补全能力”的实施路线，不走仅隐藏入口的短期止血方案；M5 借阅按正式审批流建设，不再收敛为纯直借模型。
第一阶段

主流程闭环补齐

阶段目标：消除“前端已开放但流程不成立”与“高风险写操作无正确边界”的问题，让核心链路先真实可用。

主要改造包

  • 冰冻流程补全为真实后端闭环，不再允许前端依赖 mock。
  • 技术医嘱工作站补全真实动作链。
  • 技术任务池权限拆分并强制落后端。

关键依赖

  • 复用现有 frozen_flag、病理号规则 FROZEN、诊断任务 task_type=FROZEN、报告表 frozen_diagnosis_result。
  • 工作站动作接入需同步确认后端接口、状态变化、落库与日志。
  • 权限拆分需同步角色授权与服务层二次校验。

验收标准

  • 前端冰冻工作台所有主动作均调用真实接口，移除 USE_FROZEN_WORKFLOW_MOCK = true 依赖。
  • 四类技术医嘱工作站不再出现“功能待接入”提示。
  • 无对应写权限的账号无法通过后端直接调用技术任务写接口。
第二阶段

统一追踪、统一审计、借阅审批与并发一致性

阶段目标：把“能跑”提升为“可追、可审、可控”，解决跨模块审计断链和审批语义断层。

主要改造包

  • 补申请单主对象统一流程事件。
  • 补正式报告分发事件链。
  • 整理未纳入统一权限/审计体系的敏感接口。
  • M5 材料借阅升级为正式审批流。
  • 补借阅和库存并发安全。

关键依赖

  • 申请单 create / update / void 接入统一 tracking。
  • 报告 print / issue / scheduled issue / recall 写入标准 workflow event。
  • OperatorVerificationController 与 MyNotificationController 要明确认证/授权/审计规则。
  • M5 借阅需新增审批状态、审批动作、审批角色和退回路径。
  • 归档借阅、白片借记引入版本控制、条件更新或锁策略。

验收标准

  • 申请单与正式报告分发动作可在统一 tracking 中看到。
  • 借阅从“申请”到“审批”到“借出/归还”具备清晰状态和权限边界。
  • 并发借出/归还不会出现库存负数、重复借出、位置状态覆盖。
第三阶段

状态模型、接口收敛、展示一致性与幂等治理

阶段目标：把当前“能用但容易误解、难维护”的部分收敛成稳定模型。

主要改造包

  • 重构 M2 标本与异常状态表达。
  • 收敛重复或未明显使用的接口。
  • 统一“主状态”和“分发状态”的前端展示规范。
  • 批量动作与日志正文标准化。

关键依赖

  • 补足待接收、取消、撤回、返工等关键中间态，或引入状态历史表。
  • 评估技术登记 detail 与 workspace 双轨接口，保留一条主用契约。
  • 报告内容状态与报告交付状态必须分开显示。
  • 统一批量接口回执结构、日志正文与幂等策略。

验收标准

  • 标本、报告、借阅的状态说明在前后端一致，且能解释撤回、取消、返工。
  • 重复接口减少，调用链更清晰。
  • 批量动作与审计日志可直接用于排障和复盘。

附录说明

关键接口 / 模型调整

调整对象 调整内容
冰冻流程接口族 新增真实后端接口族，输出契约以现有前端 frozen-workflow 类型为准。

并发控制 借阅与白片库存相关表增加并发控制字段或条件更新约束。

测试与验收场景

  • 冰冻流程：会话创建后可按接收、取材、切片、电话回报、报告确认推进，非法跳转会被后端拒绝。
  • 技术工作站：四类工作站关键动作可执行、可回显、可刷新，无权限用户调用写接口返回拒绝。
  • 统一追踪与审计：申请单 create / update / void 与报告 print / issue / recall 可在 tracking 与 operation log 中回放。
  • M5 借阅审批：申请、审批、借出、归还、异常登记全链路跑通，并发借阅只能成功一次。

默认假设与决策

  • 默认按“直接补全能力”推进，不采用先隐藏入口的短期止血方案。
  • M5 借阅按“正式审批流”建设，不再收敛为纯直借模型。
  • 冰冻流程优先复用现有病例、标本、诊断任务、报告表与统计口径，不单独再造一套平行领域。
  • 统一 tracking 继续以现有 workflow_events 为主，不另起第二套追踪体系，除非实现时确认无法兼容。
