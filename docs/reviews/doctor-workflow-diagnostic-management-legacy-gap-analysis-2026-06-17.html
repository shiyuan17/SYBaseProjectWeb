Doctor Workflow Legacy Gap Analysis

诊断管理现状 vs 2.31 医生组工作站对比分析

本报告对照当前前端诊断管理模块与 2.31 旧版医生组工作站，聚焦诊断主流程、状态体系、操作项与附件项目四类能力，给出管理层可读的结论、差距归因与建议优先级。报告仅基于现有代码、接口、类型与旧系统配置/源码痕迹，不构成已确认的实施需求。

范围: 全诊断模块 证据: 代码与配置 口径: 不含二进制逆向 结论: 旧系统能力未完全回补
7 当前项目已拆分的诊断管理页面：诊断分配、诊断工作台、病理报告、报告追踪、报告修订、会诊管理、病理医嘱执行。
4 固定对比维度：主流程、状态体系、操作项、附件项目。
P0 / P1 / P2 建议优先级按“闭环缺口、旧系统核心依赖、增强型体验”三层给出，不默认要求立即等价复制旧系统。

术语定义

  • 操作项：页面主按钮、行级动作、批量动作、流程推进动作、更多菜单动作。
  • 附件项目：不属于主流程状态推进，但直接支撑诊断闭环的周边能力与数据面板。
  • 页面附属链路：医嘱、会诊、修订、报告痕迹、危急值/回收申请等。
  • 报告附件能力：PACS、历史病理、申请单影像、采图/图片加入报告、打印预览、模板/样式。
分析边界 本报告只覆盖“诊断管理”模块，不扩展到送检、技术处理、归档、统计或发布流程；也不对旧系统做运行时抓取或反编译，只使用可直接检索的配置、资源命名、源码与历史变更痕迹。

执行摘要

核心判断

当前项目是模块化拆分方案

当前诊断管理已按职责拆成 7 个页面，以路由、权限和接口边界分担旧系统单页职责。这样更利于前后端契约治理，但天然弱化了旧系统“单工作台集中完成诊断闭环”的使用体验。

核心判断

2.31 是高度集成诊断平台

旧系统在 FrmDiagnosticPlatform 与 FrmRep 这一组窗体中同时承载列表筛选、报告编写、状态颜色、PACS、历史病理、图片/加入报告、会诊、修订提醒与部分医嘱联动。

管理含义

差距不只在“有没有页面”

当前项目已覆盖大部分主流程页面，但对旧系统的“集中态工作区能力”“报告附件编排能力”“提醒/回收类边缘操作”仍存在明显缺口，且这些缺口会直接影响医生的连续操作体验。

一句话结论 当前项目已经搭建出诊断管理的模块化骨架，但与 2.31 医生组工作站相比，仍缺少旧系统最有价值的几类“工作台聚合能力”：诊断列表驱动的连续处理、图片/医嘱加入报告、报告相关提醒与回收申请、以及更强的附件面板联动。

流程对比

1. 分配
  • 当前：DiagnosisAssignmentView.vue 独立完成查询、用户选择、初步分片、签发分片。
  • 旧版：诊断平台保留诊断列表导向，历史痕迹显示存在“删除主阅/解除主阅/排序/高级查询”等围绕诊断列表的深度操作。
2. 接单 / 诊断
  • 当前：DiagnosisWorkbenchView.vue 以“队列 + 报告编辑 + 材料区”三栏布局承载诊断。
  • 旧版：FrmDiagnosticPlatform 明确记录列表默认状态、颜色、分页、筛选记忆、列表颜色规则与面板尺寸配置。
3. 报告
  • 当前：诊断工作台负责草稿/初步/复核/签发；PathologyReportView.vue 负责审核、驳回、签发、发布、打印、发放、回收。
  • 旧版：诊断平台与报告工作站高度一体，存在“签发并打印”“回退报告”“批量确认报告”“报告状态颜色同步”等痕迹。
4. 追踪
  • 当前：ReportTrackingView.vue 独立提供病例生命周期、报告链区、会诊/修订/医嘱链路。
  • 旧版：更多是诊断列表内嵌历史、状态、诊断痕迹与相关 tab 的模式，而不是单独的全链路追踪页。
5. 修订
  • 当前：ReportRevisionView.vue 单独页面发起和审批修订。
  • 旧版：存在“修改修订提醒”“回退报告”“修订号”与报告状态联动痕迹，说明修订更贴近报告主工作面。
6. 会诊
  • 当前：ConsultationWorkstationView.vue 与工作台内嵌会诊 pane 双入口并存，侧重科内会诊。
  • 旧版：同时存在“科内会诊”“外院会诊”“会诊模板/材料类型/会诊医院”等更丰富痕迹。
7. 医嘱 / 附件支撑
  • 当前：工作台内嵌医嘱 pane 与独立医嘱执行页并存；附件能力分散在 tabs 与采图区。
  • 旧版：医嘱加入报告、图片加入报告、PACS、提醒、打印样式均直接贴近诊断主界面。
流程层面的主要差异 当前项目按“页面职责清晰”组织流程，旧系统按“医生一次进入一个诊断平台尽量做完所有事”组织流程。前者适合治理，后者适合高频连续作业；这正是当前体验差距的根因。

状态域 当前项目 2.31 旧系统证据 对比判断

结论：当前项目在“状态定义清晰度”上优于旧系统，但在“状态驱动的集中操作体验”上弱于旧系统。

操作项对比

页面 / 主题 当前项目可见操作项 2.31 旧系统证据项 差异结论
诊断分配 查询、重置、初步分片、签发分片；取消分片/分片清零仍为禁用占位。 旧系统痕迹显示诊断列表存在主阅相关操作，如“删除主阅”“解除主阅”。 当前具备基础分配能力，但旧系统的人员调整类细粒度操作尚未回补。
诊断工作台 暂存、初步、复核、签发，延时发放模式选择，拖拽布局，采图上传/删除。 旧系统有快捷键、状态颜色、列表高级查询、回退报告、签发并打印、更多菜单等深度操作痕迹。 当前覆盖主线，但缺少旧系统围绕“连续阅片与连续签发”的增强操作集。
病理报告 审核通过、驳回、签发、发布、打印、发放、回收。 旧系统同样有打印、签发、回退、批量确认、打印状态同步、签发后打印控制。 当前已具备正式流转闭环，但旧系统的批量化与就地回退能力更强。
报告修订 发起修订申请、审批通过、审批驳回。 旧系统有“修改修订提醒”“回退报告”与修订号痕迹。 当前流程比旧系统更干净，但缺少提醒驱动与回退联动。
会诊管理 发起会诊、录入参与人意见、完成会诊。 旧系统既有科内会诊，也存在外院会诊、会诊材料类型、会诊医院等痕迹。 当前是“科内会诊工作台”，旧系统更像“会诊综合能力”。
病理医嘱执行 查询、接收、完成、取消；工作台内嵌时还支持收费、确认收费、重新收费、提交医嘱。 旧系统有“医嘱加入报告位置”“加入报告按项目类型分行”“全部加入报告”“显示辅助检查按钮”等。 当前把医嘱执行做出来了，但旧系统“医嘱直接服务报告编辑”的特色能力缺失明显。
操作项层面的关键结论 当前项目最明显的不是“按钮少”，而是“按钮分散”。旧系统把医生高频动作贴在诊断主界面周边，当前项目则将这些动作拆入多个独立页面或子面板，连续操作成本更高。

差距归因

重构选择导致的差距

  • 当前项目按页面职责拆分，天然牺牲旧系统“一个诊断平台完成多数动作”的集中感。
  • 报告、追踪、修订、会诊、医嘱被拆成多个页面后，跨页跳转替代了同屏联动。

尚未补齐的旧系统能力

  • 医嘱加入报告、图片加入报告、全部加入报告、只看已加入报告图片。
  • 危急值提醒、回收申请、回退报告等提醒/边缘操作链路。
  • 外院会诊、会诊材料、会诊医院等更宽的会诊域能力。

当前项目的正向收益

  • 状态、接口、权限边界比旧系统更清晰，适合持续交付和测试。
  • 追踪页把病例全链路拉平展示，治理与排障价值高于旧系统散落式 tab。
  • 修订与会诊独立建模，比旧系统更适合稳定契约化。

建议优先级

P0

优先补齐直接影响诊断闭环的缺口

  • 工作台内补充“图片加入报告 / 医嘱加入报告”设计与接口方案。
  • 明确危急值提醒、报告回收申请在当前架构中的归属页与入口。
  • 补齐旧系统最关键的主界面连续操作缺口，避免医生频繁跨页。
P1

增强工作台聚合能力

  • 评估将部分报告、修订、会诊动作回收到工作台内联操作。
  • 增强报告状态颜色、列表快捷筛选、常用操作快捷入口。
  • 把申请单影像、PACS、历史病理的展示再做“医生优先”编排。
P2

谨慎吸收旧系统增强体验

  • 再评估外院会诊、回退报告、打印样式高级配置是否需要完整迁移。
  • 仅在明确业务收益时回收旧系统的复杂配置项，避免把历史配置噪声原样搬进新系统。
  • 保留当前项目的契约清晰性，不建议盲目回到单体超大工作台模式。
