1. 项目整体评价

技术选型本身没有问题。Vue 3 + TypeScript + Pinia + Vue Router + Element Plus + pnpm + turbo 适合中后台场景，工程也具备 monorepo、共享包、统一请求层、统一路由入口和测试基线。问题不在“选型错误”，而在“架构落地质量失衡”。

技术栈合理 分层执行弱化 页面层失控
  • 基础壳层清晰： apps/web-ele/src/router/routes/index.ts 使用 import.meta.glob 聚合模块路由，具备动态路由和按模块组织的基础能力。
  • 请求层集中： apps/web-ele/src/api/request.ts 已统一挂载鉴权头、默认响应拦截器和错误提示。
  • 架构问题不在壳层，而在业务实现： 目录看似模块化，但复杂度没有沉到 composable、store action、服务编排层，而是大量回流到视图文件。
  • 定性判断： 这不是典型的“架构混乱型项目”，而是“架构框架正确，但执行纪律不足，页面层复杂度长期无人兜底”的项目。
结论：项目具备现代前端工程的基础壳层，但尚未形成足以约束业务复杂度的执行机制。当前代码库仍可维护，但维护成本已经进入上升区间。

2. 代码质量

当前代码质量的核心问题不是“局部写法差”，而是“超大 SFC + 页面承担过多职责”。这已经不是审美问题，而是明确的维护性风险。

单文件复杂度过高 视图层承担业务编排 类型边界偏宽

结论

  • 组件职责边界不稳定，多个页面已经同时承担权限判断、请求编排、状态机、弹窗流转、格式化和跳转逻辑。
  • 目录层面的模块化存在，但实现层面的复用与职责下沉不足，导致复杂度集中在页面文件本身。
  • TypeScript 基线可用，但并不严格，宽泛对象类型和兜底断言仍然存在。

3. 性能与体验

首屏与路由层具备现代实践基础，主要风险不在构建链，而在运行期的页面复杂度、重复刷新编排和大组件状态同步成本。

支持懒加载 页面状态链条过长 重复请求治理不足

结论

  • 动态路由和页面懒加载已使用，基础首屏策略并不落后。
  • 实际风险来自页面运行期：局部状态多、弹窗多、明细区多、watch 链多、表格操作重。
  • 当前没有直接证据证明存在确定性内存泄漏，但复杂页面结构已经显著提高了卡顿和泄漏概率。

证据

  • 路由层具备懒加载基础： apps/web-ele/src/router/routes/index.ts 通过 import.meta.glob 聚合模块路由。
  • 大型页面内部存在高频状态联动： SpecimenManagementView.vue 中多个弹窗、抽屉、列表、详情和路由回填逻辑共同存在； ArchiveManagementView.vue 同时维护归档柜、柜位、借还、记录查询等多块工作台状态。
  • 请求并发控制未形成明显统一策略： 页面级查询多以 loadXxx() 方式直接触发，报告中未看到通用的取消请求、过时结果丢弃或最新请求优先封装。
  • 图表/监听类页面需重点关注： 项目中存在多处 watch 调度，当前尚不足以断定泄漏，但足以判定复杂度持续上升。
这里不能下“性能差”的武断结论。更准确的表述是：项目已经出现明显的性能脆弱性，且后续新增需求会优先放大这一脆弱性。

5. 安全与边界

安全部分必须直接下风险判断。当前项目没有出现明显的恶意代码迹象，但存在若干不能轻描淡写的边界问题。

存在 XSS 注入面 认证恢复能力偏弱 持久化策略需复核

主要风险

  • XSS 风险面存在： 共享包内存在直接 v-html 渲染。若上游内容没有严格净化，组件本身就形成注入入口。
  • 认证恢复能力弱： 请求层统一注入 Authorization 头，但刷新 token 被显式禁用。
  • 会话持久化策略需复核： 看到共享 store 的持久化插件和本地存储使用，但未在本次审查中拿到完整安全策略说明。
  • 路由权限依赖首次访问流程： 权限守卫虽然没有塞入大量业务请求，但权限生成仍强依赖 isAccessChecked 与首次加载流程。

证据

  • XSS 注入面： packages/effects/common-ui/src/ui/dashboard/workbench/workbench-trends.vue:51、 packages/effects/common-ui/src/ui/dashboard/workbench/workbench-todo.vue:50、 packages/effects/plugins/src/tiptap/preview.vue:32 使用 v-html。
  • 认证恢复关闭： apps/web-ele/src/api/request.ts:46 明确抛出 Refresh token is disabled； apps/web-ele/src/api/request.ts:57 统一注入鉴权头； apps/web-ele/src/api/request.ts:87 主要依赖全局 ElMessage.error 提示。
  • 权限守卫路径： apps/web-ele/src/router/guard.ts:90-98 使用 isAccessChecked 和 Promise.all([fetchUserInfo, getAccessCodesApi]) 生成访问能力。
  • 本地持久化迹象： packages/stores/src/setup.ts:57 涉及 localStorage； packages/stores/src/modules/access.ts:102 存在 persist 配置。
这里最需要避免的误判是“现在没出安全事故，所以暂时不是问题”。v-html 和会话持久化边界都属于静态审查阶段就应明确标红的事项。

6. 高优先级改进建议（4 条）

以下建议按“问题 → 风险 → 方案”组织，优先级已经按实际治理收益排序。

优先级 问题 风险 具体改进方案
P1 超大页面长期堆积 单个页面成为事实上的微型子系统，任何需求变更都可能引发跨区域回归。 先从 ArchiveManagementView.vue 和 SpecimenManagementView.vue 开始拆分， 以“列表筛选区 / 明细区 / 对话框流 / 数据编排”拆成多个 section component 和 composable， 禁止再向原页面继续堆叠职责。
P1 请求与状态编排滞留在视图层 页面直接承担业务流程后，错误处理、并发策略、刷新策略和复用都会碎片化。 将页面级 loadXxx、提交、重试、路由回填、数据拼装下沉为 useXxx composable 或模块 store action， 页面只保留装配职责和极薄的交互入口。
P2 路由路径与权限码硬编码散落 路径或权限变更需要全仓搜改，且模板内字符串难以被重构工具稳定发现。 建立模块级路由常量与导航 helper，逐步替换模板内直接写死的 path 和业务权限字符串；为高频工作台页优先治理。
P1 安全边界与类型边界都偏松 v-html 形成注入面，宽泛对象类型削弱关键数据约束，问题一旦叠加会放大事故面。 对所有 v-html 引入明确的上游净化责任或替代渲染方案； 收紧 any、Record<string, any>、Recordable<any> 的使用范围， 先从认证表单、共享适配层和表单基础设施治理。

7. 验证与边界说明

已验证

  • 已检查根工程与 apps/web-ele 的工程结构、依赖与脚本。
  • 已抽样审查请求层、路由守卫、认证 store、共享包风险点和代表性业务页面。

  • 已记录 1 条 Vitest mock 顶层提升警告，属于后续兼容性风险。

未验证

  • 未执行真实生产流量下的性能剖析，无法对首屏时长、交互耗时和内存曲线下最终结论。
  • 未完成浏览器专项兼容矩阵验证，尤其是导出、打印、国产环境与复杂工作台页面。
  • 未对后端真实 token 生命周期、CSRF 策略、服务端净化责任进行联调确认。
  • 未进行渗透测试或动态安全测试，本报告不替代安全专项审计。
最终判断： 这是一个“工程基础不差，但业务复杂度治理明显滞后”的项目。最优先的工作不是继续堆功能，也不是先大规模改工具链，而是压制页面层膨胀、收回业务编排边界、补齐安全与类型约束。继续按当前模式扩展，未来的主要成本不会是开发速度，而是回归成本和修改不确定性。
