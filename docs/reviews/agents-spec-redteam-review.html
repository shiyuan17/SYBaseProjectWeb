2. 详细发现

高 R-01 · 治理规则缺乏机器门禁，可被静默绕过

类别：规则不起作用（红队核心关切）

规范要求大量“硬约束”：任务开始必须输出任务确认模板（AGENTS §4）、PR 必须填写 Workflow Packet 与 Memory Update Packet（AGENTS §7、DYNAMIC §Workflow Packet）、高风险必须做 Red Team、红区必须人工确认（AGENTS §5/§6）。但实际门禁只有：

lefthook.yml（pre-push）
check-type → pnpm check:type，check-circular → pnpm check:circular，test-unit → pnpm test:unit
.github/workflows/frontend-quality.yml
check:type · build:ele · test:unit · lint · check:circular（无 PR 正文/Packet 校验，无 cspell/dep/e2e）
docs/CODING_RULES.md:63 / package.json:24
cspell lint "**/*.{ts,tsx,vue}" "**/README.md" —— 明确不扫描 docs/*.md 规范与根目录五类记忆文件

红队攻击路径：一个 PR 完全不写 Workflow Packet、不做 Red Team、不更新记忆文件、甚至在 AGENTS.md 里引入错别字或矛盾条款——只要类型/循环依赖/单测/lint 通过，CI 与 hook 都会放行，无任何拦截。即“规则写了，但没有任何执行回路保证它被遵守”。

影响：这是全套规范最系统性的弱点：治理强度等同于“君子协定”，对不合作或疏忽的提交者无效，长期会导致规范与现实漂移。

建议：(1) 增加一个轻量 CI job 校验 PR 正文是否含必填 Packet 字段（关键词存在性即可）；(2) 将 cspell 扫描范围扩展到 docs/**/*.md 与根目录记忆文件，或新增 markdown lint；(3) 在 AGENTS.md 显式承认“本规范以软约束为主、机器门禁仅覆盖 X/Y/Z”，避免读者高估强制力。

中 R-02 · “Issue 模板”孤儿引用

类别：引用不当（指向不存在的产物）

AGENTS.md §9 语言与提交约束
“…及 .github PR/Issue 模板沿用其现有英文结构…”

核验：.github/ 下仅有 PULL_REQUEST_TEMPLATE.md 与 workflows/，不存在 ISSUE_TEMPLATE（已用 glob 确认 0 命中）。

影响：读者会误以为存在 Issue 模板并去“沿用”，造成困惑；也可能掩盖“本项目用 Linear 而非 GitHub Issue”的真实流程。

建议：改为“.github PR 模板”，或补一个真实的 ISSUE_TEMPLATE；若 Issue 流程在 Linear，应明示。

中 R-03 · 主 Workflow 名称 Workflow-Infra / Workflow/Infra 混用

类别：命名冲突 / 不一致

DYNAMIC_WORKFLOW_RULES.md:17 / :299 / :307
“…Production Debug、Workflow/Infra 之一”；章节标题 ## Workflow/Infra Workflow
DYNAMIC:37 · AGENTS.md:101 · PULL_REQUEST_TEMPLATE.md:14
触发信号表 / 任务确认模板 / PR 模板均写作 Workflow-Infra（连字符）

影响：同一枚举项出现两种写法，破坏“同一概念全项目统一命名”（CODING_RULES §1）；若未来做 PR 正文自动校验或脚本匹配，会因斜杠/连字符不一致而漏匹配。

建议：统一为一种（推荐连字符 Workflow-Infra，与表格/模板一致），同步修正总规则与章节标题。

中 R-04 · 红区清单重复定义且不一致

类别：重复编写 + 内容漂移

AGENTS.md §5 文件操作边界（红区）
权限模型、登录态、全局拦截器、构建配置、发布脚本、浏览器兼容兜底方案
GIT_RULES.md §7
“涉及红区文件（权限模型、登录态、全局拦截器、构建配置、发布脚本）…” —— 缺少“浏览器兼容兜底方案”

影响：红区是风险分级的核心定义，被两处各自枚举一份。GIT_RULES 漏掉一项，已经发生事实漂移；以后只改一处会进一步扩大分歧。

建议：红区定义只在 AGENTS.md §5 维护，GIT_RULES 改为“见 AGENTS.md §5 红区定义”而不重列具体条目（与 §8 记忆层“唯一来源”做法保持一致）。

中 R-05 · 修饰器名称中英混用

类别：命名不一致

AGENTS.md §4 / DYNAMIC 触发信号表
强制修饰器：Security / DB / Red Team / Backend Cross-check / Browser 验证
PULL_REQUEST_TEMPLATE.md:17
Required modifiers: Security / DB / Red Team / Backend Cross-check / Browser Verification

影响：同一修饰器中文档叫“Browser 验证”、PR 模板叫“Browser Verification”，填写者难以一一对应，弱化可核对性。

建议：固定一种写法（PR 模板英文 + 文档中文括注，如 Browser 验证（Browser Verification）），全链路统一。

低 R-06 · 顶层文档清单在 4 处重复维护

类别：重复编写 / 漂移风险

同一份顶层规范文档列表分别出现在：AGENTS.md §1 必读顺序、AGENTS.md §2 规范映射表、AGENTS.md 关联文档、docs/README.md 顶层规范文档。新增/重命名一个规范文档需同步改 4 处。

建议：以 docs/README.md 为清单单一来源，AGENTS 其余处尽量引用而非复制完整列表（必读顺序确有排序价值可保留，但关联文档段可精简）。

低 R-07 · CODING_RULES 内部重复描述 cspell 范围

类别：重复编写

CODING_RULES.md:63
“…cspell 校验（扫描 **/*.{ts,tsx,vue} 与 **/README.md，不覆盖其他 *.md）”
CODING_RULES.md:69
“pnpm check:cspell 扫描 **/*.{ts,tsx,vue} 与 **/README.md，仍不覆盖其他 *.md…”

建议：合并为一处描述，避免两行近乎同义、未来各改一半。

低 R-08 · AI-CODE-HEALTH 与专项规范的软冲突

类别：潜在冲突（已被优先级规则形式化化解）

  • any：AI-CODE-HEALTH §4.1“禁止 any，必须用 unknown”；VUE_TS_RULES §3 / AGENTS 健康基线只说“避免滥用 any”。硬禁 vs 软避。
  • 行数：AI-CODE-HEALTH §2.5“UI 组件 ≤100 行”；VUE_TS_RULES“单个页面组件超过数百行…必须拆分”。阈值不同。

缓解：AI-CODE-HEALTH §使用边界已明确“专项规范 > 本规范”，且自述行数为“参考值不是目标”，冲突在形式上已被消解。但读者仍会因两套数字产生认知负担。

建议：在 AI-CODE-HEALTH 冲突点就地加一句“具体口径以 VUE_TS_RULES 为准”，降低误读。

低 R-09 · 跨仓引用缺仓库限定符

类别：引用易误读（路径本身存在）

DYNAMIC_WORKFLOW_RULES.md:161
“…对照后端 docs/rules/DB_RULES.md 与迁移脚本”

核验：后端 SYBaseProject/docs/rules/DB_RULES.md 确实存在，路径正确。但在前端仓库语境下写 docs/rules/... 容易被误当成本仓库路径（本仓库 docs/ 下并无 rules/）。

建议：加仓库前缀，如 SYBaseProject/docs/rules/DB_RULES.md，与 AGENTS §3“不写死绝对路径、但需明确跨仓”精神一致。

低 R-10 · 门禁分布不对称，易生“hook 过=安全”误解

类别：门禁覆盖说明

实际门禁分布：lint 仅在 CI（不在 pre-push）；cspell / check:dep / test:e2e 既不在 pre-push 也不在 CI（仅 pnpm check 与 RELEASE §5 发布门禁覆盖）。GIT_RULES §7 已部分说明“hook 不覆盖完整构建/E2E/联调”，缓解了大部分风险，但未点明 lint 不在 pre-push、cspell 不在 CI 这类细节。

建议：在 GIT_RULES §7 或 CODING_RULES 补一张“门禁覆盖矩阵”（命令 × pre-commit / pre-push / CI / 发布门禁），让“哪层拦哪些”一目了然。

3. 良好实践（建议保留，不要在整改中破坏）

逐项对账通过：13 个顶层规范文档、lefthook.yml 的 4 个 hook 与具体命令、frontend-quality.yml 步骤、tests/e2e/smoke/auth-router-request.spec.ts、package.json 中 dev:ele/lint/check:type/check/test:unit/test:e2e/build/build:ele/check:circular/check:cspell/check:dep 脚本、engines 字段、.logs/* 写入行为，均与文档描述一致，未发现指向不存在脚本/文件的硬性失效引用。
