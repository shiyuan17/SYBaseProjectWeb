Technical Workflow Gap Analysis

制片管理现状 vs 旧技术组工作站对比分析

本文档以当前仓库 technical-workflow 模块为“制片管理”现状真值， 以参考文件 technical-workstation.html 中描述的旧技术组工作站为对照基线， 聚焦回答三件事：流程有什么不一样、要素有什么区别、交互有什么区别， 并明确当前 M2 与 M3 的职责边界。

当前基线：M3 technical-workflow 旧链路整体纳入对比 登记/接收已外移到 M2 以当前前端页面现状为主口径
9 个页面 当前制片管理由入口页、任务池、5 个主工位、返工和追踪页共同组成，而不是单一工作站页面。
2 条主判断 当前 Web 新增了任务池、入口总控、脱水批次、返工与追踪；旧工作站内置的登记/接收已前移到 M2。
3 类差异 差异统一归纳为外移、显式化/新增、以及交互模式重组，避免把边界变化误判为功能缺失。

1. 背景与结论摘要

旧技术组工作站是典型的桌面端流水线工作站模型：从登记、接收到取材、包埋、切片、染色出片， 通过多个 WinForms 页面直接推进病例。当前 Web 端的“制片管理”则只覆盖 M3， 默认前提是病例已经完成 M2 的接收上游流程，再进入技术生产线。

核心结论一
  • 当前 Web 比旧 HTML 多了任务池、入口总控、显式脱水批次、显式返工、技术追踪。
  • 这些不是简单“页面装饰”，而是把生产调度、异常闭环和上下文查看前移成了一级能力。
核心结论二
  • 当前 Web 比旧 HTML 少了技术组内自带的登记/接收入口。
  • 这不是当前制片管理遗漏，而是业务边界调整：登记/接收已经前移到 M2，不再属于当前 M3 模块本体。
M2 / M3 边界 当前体系下，M2 承接申请、标本登记、固定、转运、接收等上游动作； M3 从技术待办进入，负责取材、脱水、包埋、切片、染色、返工和技术追踪。 因此旧工作站中的“登记 / 接收”保留在本报告中参与整体链路对比，但结论必须标记为“已外移”，而不是“当前缺失”。

3. 流程差异对比

旧技术组工作站强调“技术组内部一站式流水线”，当前 Web 强调“上游完成接收后，进入任务化制片工作台”。 因此两者的主差异不是某个工序消失，而是起点、组织方式和闭环能力发生了重组。

旧整体链路

登记 → 接收 → 取材描写 → 包埋 → 切片 → 染色出片

  • 登记与接收都属于技术组工作站内部入口。
  • 取材后直接衔接包埋、切片和染色。
  • 常规切片与冰冻切片共用切片能力，但以不同入口进入。
  • 流程推进主要依赖多个窗体切换与现场扫码/打印动作。
当前 Web 链路

M2 接收上游 → M3 入口页 / 任务池 → 取材 → 脱水 → 包埋 → 切片 → 染色 → 返工 → 技术追踪

  • 登记/接收不再位于制片管理内部，而是作为 M2 前置。
  • M3 以入口总控和任务池作为统一调度起点。
  • 新增显式“脱水”批次环节，不再在旧链路中隐式承接。
  • 返工和技术追踪被抬升为独立页面，而不是散落在各工位后的附属动作。
流程节点 旧技术组工作站 当前制片管理 差异结论 说明
登记 技术组内部入口，工作站起点。 不在当前 M3 内，已前移到 M2。 外移 当前制片管理不再负责病例首次登记，这是边界调整，不是前端缺失。
接收 登记接收作为技术组内部流程节点。 不在当前 M3 内，已前移到 M2 标本接收链路。 外移 当前进入制片管理时默认已完成接收上游。
取材描写 通过 FrmPlan / 取材台直接进入处理。 通过任务池或工位队列进入 /technical-workflow/grossing。 显式化 工序仍在，但入口从“模块按钮直达”改为“任务驱动 + 工位化处理”。
脱水 旧参考主流程里未被单独强调为工作台层级。 当前有独立 /technical-workflow/dehydration，强调批次创建、开始、完成。 新增显式化 当前 Web 把原本偏后台或隐式的制程步骤抬成了明确工位。
包埋 独立包埋页面。 独立包埋工位，且带病例上下文与下一工位提示。 显式化 工序一致，但当前 Web 更强调流程上下文与衔接关系。
切片 常规切片和冰冻切片分入口，共用切片处理能力。 当前统一到 /technical-workflow/slicing，未拆出独立冰冻入口。 弱化 当前前端重点是统一切片工位模型，未把冰冻流程显式展开为单独 Web 工位。
染色出片 独立染色页面，承接切片后的上机出片。 独立染色工位，并与追踪、返工、异常提醒联动。 显式化 当前 Web 增强了与后续追踪、异常闭环的连接。
返工 旧参考只在流程说明中提及返工评价，不是明确独立入口。 独立 /technical-workflow/rework，支持创建、执行、回流建议。 新增显式化 返工从隐式异常处理变成了明确的闭环中心。
追踪 旧参考以流程说明和上下游关系理解为主。 独立 /technical-workflow/tracking，聚合对象树、时间线、任务、返工、QC。 新增显式化 当前 Web 为制片线增加了专门的可视化追踪层。

4. 要素差异对比

旧工作站和当前制片管理的区别，不只在页面名字，更在于页面承载的对象、状态、异常、角色和批次能力不同。 下表按固定矩阵逐项对比。

维度 旧技术组工作站 当前制片管理 差异结论 说明
页面入口形态 主窗体按钮直达多个独立 WinForms 页面。 根路由下组织为入口页、任务池、工位页、返工页、追踪页。 合并承载 当前入口更像“工作台集合”，而不是同级按钮菜单。
工序节点 登记、接收、取材、包埋、切片、冰冻、染色。 M2 接收上游 + M3 取材、脱水、包埋、切片、染色、返工、追踪。 外移 + 新增 登记/接收外移到 M2，脱水/返工/追踪在 M3 中被显式抬升。
任务对象 以申请单、标本、蜡块、玻片为主，更多依赖页面语义理解。 类型显式分层为病例、标本、蜡块、包埋盒、玻片、技术任务、返工单。 显式化 当前前端通过任务对象和追踪对象模型把实体层级直接展示出来。
状态/追踪信息 状态推进分散在各页面和后台逻辑中。 任务状态、当前节点、超时、病例状态、事件轨迹、对象树集中可见。 显式化 当前 Web 强调状态可视和追踪可查。
批次能力 旧参考中更强调工序执行本身。 脱水批次被明确建模，支持创建、开始、完成和批次反馈。 新增显式化 批次处理在当前 Web 中是重要的一类对象和动作。
返工能力 返工评价存在，但不是单独工作台主视角。 返工单支持创建、执行、回到对应工位和下一步建议。 新增显式化 返工从“后置异常动作”变成“主流程闭环能力”。
质量/异常信息 更多依赖现场经验、扫码和页面内反馈。 任务超时、质控问题、返工提醒、异常模式入口被统一表达。 显式化 当前 Web 更强调异常聚合和风险前置。
设备/扫码/打印表达 扫码、打印、现场设备感更强，是高频交互中心。 保留终端编码、打号结果、附件占位等表达，但设备集成更多是字段和结果反馈层。 弱化 当前前端更偏工作台组织，设备交互深度弱于旧桌面端现场形态。
角色分工表达 从页面名称和岗位操作隐式体现登记员、接收员、取材技师、切片技师、染色技师。 通过权限码、任务分派、责任技师、工位权限和回流建议来表达角色分工。 显式化 当前 Web 的角色边界更依赖任务与权限，而不是单纯页面名称。

5. 交互差异对比

旧技术组工作站强调“模块按钮直达 + 现场高频处理”，当前制片管理强调“任务驱动 + 工位同屏处理 + 右侧上下文”。 两者并非谁多谁少，而是交互重心不同。

旧桌面工作站交互
  • 模块按钮直达，用户先决定进哪个工作站。
  • 页面分散，流程推进依赖窗体切换。
  • 扫码、打印、批量处理是现场高频入口。
  • 同一病例的上下文更多依赖操作者记忆和页面间跳转。
当前 Web 工位化交互
  • 任务先进入入口总控和任务池，再流向具体工位。
  • 工位统一采用左侧队列、中区处理、右侧病例上下文。
  • 通过 query 深链直接把 taskId / caseId / objectId / mode 带入工位。
  • 返工与追踪被前移成显式入口，减少异常处理散落在各页。
当前交互能力 当前页面表达 相对旧工作站的变化 差异结论
entry 总览和快捷入口 /technical-workflow/entry 展示工位汇总、快捷入口、异常待处理。 旧工作站没有单独的入口总控页，更多是主窗体按钮分发。 新增显式化
tasks 分派/释放/进入工位/追踪 /technical-workflow/tasks 提供任务筛选、责任技师分派、释放责任人、进入工位、查看追踪。 旧工作站更像工序直达，不强调先在统一任务池调度。 新增显式化
统一工位骨架 取材、脱水、包埋、切片、染色页面统一为“左侧队列 + 中区处理 + 右侧上下文”。 旧页面更偏单工位功能页，缺少统一工作台骨架。 合并承载
病例上下文右侧常驻 工位中常驻病例概况、对象进度、提醒、下一工位建议。 旧工作站上下文更多通过页面切换或业务经验理解。 显式化
tracking 聚合追踪 /technical-workflow/tracking 聚合对象树、流程时间线、任务、返工、QC、事件。 旧工作站未把这些追踪信息集中成单一 Web 视图。 新增显式化
rework 创建、执行、回流建议 /technical-workflow/rework 支持从深链对象发起返工、执行返工单、回到对应工位。 旧工作站虽然有返工语义，但没有如此显式的异常闭环入口。 新增显式化
设备现场感 当前页面保留终端编码、打号结果、批次结果、附件占位等交互。 旧桌面端更强调扫码枪、打印机、现场设备的直接驱动。 弱化
交互层面的真正变化 当前 Web 不是把旧页面原样搬到浏览器，而是把“先选工作站，再在页面内找对象”重构成 “先看任务和病例上下文，再进入当前对象处理”。因此阅读差异时，应该优先理解交互组织方式变化，而不是只按旧页面名找一一对应关系。

6. 结论与建议

流程结论
  • 当前制片管理不是旧技术组工作站的“少量缺页版”，而是边界重组后的 M3 工作台。
  • 登记/接收外移到 M2，脱水、返工、追踪在 M3 中被显式增强。
要素结论
  • 当前 Web 的核心要素从“页面按钮”转向“任务对象、状态、病例上下文、异常闭环”。
  • 对象层级和追踪信息的可视化程度高于旧参考。
交互结论
  • 当前 Web 强于调度、追踪和闭环表达，弱于旧桌面端的现场设备交互感。
  • 这是工作台模型切换，不应直接解读为实现不足。
建议一：对外沟通口径 在评审或汇报中，应把当前制片管理描述为“基于 M2/M3 分层后的技术生产工作台”，而不是“旧技术组工作站 Web 版”。
建议二：后续差异判断原则 若后续有人继续追问“为什么没有登记/接收入口”，应直接说明这是 M2 负责范围。 若追问“为什么没有独立冰冻切片入口或更强设备交互”，那才属于当前 Web 相对旧桌面模型的弱化点。
建议三：需求分流 未来若要进一步贴近旧桌面端体验，优先补的是： 1. 冰冻切片等特殊流程的显式工位表达； 2. 更强的扫码、批量处理和设备反馈； 3. 工位内更结构化的异常与质量控制提示。 这些属于新的产品增强需求，不属于本次文档对齐范围。

文档假设：“制片管理现状”仅指当前仓库 technical-workflow 模块，不代表线上运行态，也不默认等价于历史桌面端完整边界。
