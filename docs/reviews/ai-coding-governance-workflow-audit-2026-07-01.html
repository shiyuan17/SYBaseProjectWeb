一、第一性原理分析

规范存在的根本目的不是“让流程显得专业”，而是降低协作系统的熵。中大型项目里，需求、代码、接口、数据库、测试、发布、人员和 AI Agent 同时变化，任何一个环节没有清晰边界，就会把局部不确定性扩散为全局风险。

对 AI Coding 来说，规范还承担第二个职责：把“隐性团队经验”转成可机器执行、可审查、可验证的约束。AI 最容易犯的不是语法错误，而是基于片段上下文编造接口、误解业务状态、扩大修改范围、跳过后端证据、把验证写成愿望。因此，一个合格的规范体系必须回答五件事：现在该读什么，哪些地方不能碰，碰了要谁确认，如何证明成功，如何把结果交给下一位人或 Agent。

当前项目已经建立了较强的治理骨架：AGENTS.md 明确了入口、三档风险、红区确认、后端联动和 Memory 规则；DYNAMIC_WORKFLOW_RULES.md 用 Workflow 和修饰器把任务风险转成证据要求；GIT_RULES.md 定义了 worktree、PR、AI 主动提交和 hook；TASK_LIFECYCLE_RULES.md 定义了 Clarify 到 Retrospective 的生命周期产物。这些设计方向是正确的。

底层风险在于：框架完整不等于闭环可靠。当前规则仍有三类断点。第一类是“规则和自动化不一致”，典型表现是 PR 模板与校验器字段结构不匹配。第二类是“关键专业规则过度合并”，API、DB、测试、Review 都有提到，但缺少足够细的契约和检查清单。第三类是“状态可追踪但不够可运营”，backlog 和任务文件存在，但状态转移、阻塞原因、验证证据和交付日志没有被统一校验。

当前规范真正解决了什么

  • 阻止 AI 在没有入口阅读和风险判定时直接编码。
  • 把红区、权限、接口、患者报告、构建发布等场景升级为 Full。
  • 要求跨仓接口和业务规则对照后端证据。
  • 用 Memory 文件承接长期业务契约，避免会话上下文丢失。
  • 用 hook、CI、PR Packet 提供基础自动化约束。

当前流程仍没有完全解决什么

  • API 变更如何形成可审查的契约、兼容策略和错误码说明。
  • 数据库迁移、回滚、种子、数据修复的前后端证据格式。
  • 单测、集成、E2E、冒烟、回归之间的强制触发矩阵。
  • Review 如何按严重度、证据质量、AI 产出风险进行操作。
  • 任务从 todo 到 done 的可审计状态流和验收证据沉淀。

二、当前规范与工作流问题清单

P0：必须立即修复的问题

问题描述 可能后果 触发场景 优先级 建议处理方式
PR 模板与 PR Packet 校验器结构不一致。模板使用 ## Memory、## Evidence、### Full Evidence，但 scripts/validate-pr-packet.mjs 对 Full Packet 仍要求顶层 ## Dynamic Tests 与 ## Memory Update Packet。 高风险 PR 可能被错误阻塞，团队可能绕过校验；也可能为了通过校验复制旧结构，导致规范正文、模板、CI 三套口径分裂。 Security、DB、Backend Cross-check、Browser Verification、红区确认、生产问题等 Full Packet PR。 P0 二选一立即统一：推荐修改校验器以支持当前模板的 Memory 和 Evidence 结构，并补测试；或把 PR 模板回滚到校验器要求的顶层结构。修复后用一份 Security Full PR fixture 跑通。

P1：短期需要优化的问题

问题描述 可能后果 触发场景 优先级 建议处理方式
缺少独立 API 规范。FRONTEND_RULES.md 提到统一 Axios、分页、mapper、错误处理，但没有接口命名、版本兼容、DTO 变更、错误码、幂等、废弃策略、跨仓证据模板。 AI 容易编造接口路径或字段，前后端字段漂移，接口签名随需求随意变化，测试只覆盖当前页面而不覆盖契约。 新增 service、mapper、mock、联调、统计口径、权限码、导出接口。 P1 新增 docs/rules/API_RULES.md，定义请求/响应/错误/兼容/证据模板，并在 AGENTS、QUICKSTART、DYNAMIC_WORKFLOW、PR 模板中引用。
数据库规范只有参考资料入口，没有前端仓的 DB 变更协作协议。当前 docs/reference/database/README.md 是资料索引，不是 migration、回滚、种子、数据修复、字段生命周期规则。 涉及后端 migration 时，前端只能靠口头或 memory 片段判断字段状态；AI 可能误把 SQL 资料当成当前生产契约。 字段新增、字段重命名、权限种子、统计指标种子、数据修复、回滚方案。 P1 新增轻量 DB_RULES.md，定位为“前端仓跨 DB 协作协议”，明确以 sibling backend 为事实来源，要求 migration ID、回滚策略、测试数据、兼容窗口、前端字段落点。
测试规范散落在 CODING_RULES.md、FRONTEND_RULES.md、RELEASE.md 和 E2E README 中，缺少一张按变更类型决定测试层级的强制矩阵。 开发者和 AI 会用“跑了一个相关单测”替代关键链路验证；E2E、冒烟、回归、手工验收的边界不稳定。 缺陷修复、权限/路由、导出打印、跨仓接口、生产问题、发布前验证。 P1 新增 TEST_RULES.md，定义单测/集成/E2E/冒烟/回归触发条件、测试数据策略、失败产物、可接受未验证声明格式。
Review 规范不足。GIT_RULES.md 有 PR 评审规则，但没有项目级 review checklist、严重度标签、AI checker 角色、证据质量判断标准。 PR Packet 字段非空但证据空洞时可能被放行；高风险 AI 产出缺少 maker/checker 分离。 高风险 PR、子 Agent 产出、共享层重构、接口/权限/患者报告/发布变更。 P1 新增 REVIEW_RULES.md，承接 correctness、architecture、security、performance、verification 五轴，要求 P0/P1/P2 发现分级和阻塞条件。
验证基线存在“长期带病运行”迹象。TECH_DEBT.md 多条 Open 项显示 full pnpm lint 被既有格式或脏改动阻塞。 “不通过测试不得交付”会变成例外驱动；AI 容易把未跑全量验证合理化，reviewer 难以判断当前失败是旧问题还是新回归。 任何需要交付、发布、合并、重构、跨层变更的任务。 P1 建立“质量基线恢复”小任务，先让 pnpm lint 和 pnpm check:type 在主分支恢复绿，再允许针对性例外。例外必须绑定 TD ID、失败摘要和影响范围。
任务状态模型不足。backlog.json 有 id、title、status、dependencies、scope，但缺少 owner、risk、packet tier、validation、blocked reason、done evidence；当前多项任务仍为 todo。 新人或 AI 可以看到任务列表，但无法判断哪些在做、哪些阻塞、哪些已验证、哪些待 review。 多 Agent 并行、跨天续接、计划拆解、验收收口。 P1 扩展本地任务 schema 或新增状态日志，最少加入 status 枚举、risk、validation、handoff、updatedAt，并用脚本校验 backlog 与任务文件一致。

P2：中长期改进的问题

问题描述 可能后果 触发场景 优先级 建议处理方式
ARCHITECTURE.md 承载大量业务契约，文件已经很长，模块契约、接口事实、状态口径混在同一快照里。 长期会变成“大型记忆文件”，AI 读取成本上升，局部模块契约难以维护，过期片段不易发现。 M2-M6 持续迭代、跨仓契约变更、统计指标和权限扩展。 P2 保留 ARCHITECTURE.md 为索引和全局边界，把模块契约逐步拆到 docs/contracts/<domain>.md 或模块 README，并由 architecture 指向。
治理文档概念较多：Fast Path、Lightweight、Full、Workflow、Modifier、Packet、Lifecycle、Loop、Memory、Worktree。虽然各有职责，但新 Agent 学习成本高。 低风险任务可能被流程压重，高风险任务可能漏读某个专项文件。 新人入场、非 Codex 工具接入、临时修复任务。 P2 新增一页 docs/rules/WORKFLOW.md 或把 QUICKSTART 扩成“决策树”，用 10 个问题完成路由，不重复正文规则。
PR Packet 校验只检查字段结构和部分非空，不检查证据质量，这是合理边界，但目前 reviewer 责任没有被 REVIEW_RULES.md 承接。 “Attack result: 已检查”这类空泛证据可能通过格式校验，质量仍靠个人经验。 Full Packet、Red Team、跨仓、发布、生产问题。 P2 把“证据质量审查”写成 Review 清单，并在 PR 模板保留 checkbox：真实命令、真实截图/接口、真实失败点、真实剩余风险。
历史审查 HTML 很多，部分新报告未进入 docs/reviews/README.md，HTML 可读性强但不利于机器提取摘要。 AI 可能漏读最新审查结论，团队难以检索“当前有效结论”和“历史档案”。 审查报告、验收报告、差距分析持续增加。 P2 约定每份 HTML 报告配一个 20 行以内 Markdown 摘要或在 reviews README 建立“当前有效 / 历史归档”分组。
红区人工确认目前主要靠文本记录，缺少确认来源格式和可审计字段。 后续 reviewer 难以判断是谁确认、确认了什么范围、是否包含回滚和验证。 权限、登录态、Axios 全局层、构建发布、环境变量、跨仓契约。 P2 把红区确认扩展为固定字段：确认人、时间、范围、原因、验证、回滚、范围是否扩大。

三、推荐的规范体系结构

不建议把当前体系推倒重来。当前项目已经有良好的 AGENTS.md + docs/rules + docs/memory 分层。推荐做法是：保留入口和主干，补齐缺失专业规则，减少重复定义，把能自动化的字段接入脚本。

推荐文件 职责 与现有文件关系 避免混乱的边界
AGENTS.md AI 协作入口：读什么、风险、红区、交付、Memory 总规则。 保留为最高入口。 只放决策入口和强制边界，不堆 API/DB/Test 细则。
docs/rules/QUICKSTART.md 最小阅读路径和场景路由。 保留，补引用新增 API/DB/TEST/REVIEW。 只回答“这次最少读什么、跑什么”。
docs/rules/WORKFLOW.md 或保留现有两文件 统一解释 Clarify 到 Handoff 与 Dynamic Workflow 的关系。 可作为 TASK_LIFECYCLE_RULES.md + DYNAMIC_WORKFLOW_RULES.md 的轻量索引，不必替代它们。 不要新增第三套 Packet 字段。
docs/rules/API_RULES.md API 命名、DTO/ViewModel、错误码、分页、兼容、废弃、mock、后端证据。 从 FRONTEND_RULES.md 拆出接口细则；FRONTEND 只保留摘要。 不替代后端 API 文档，只规定前端如何消费和变更契约。
docs/rules/DB_RULES.md 跨仓 DB 变更协作协议：migration、seed、回滚、数据修复、字段兼容、证据。 引用 sibling backend 的 DB 规范和 migration 路径。 前端仓不维护真实 DB 规则全文，只维护协作要求。
docs/rules/CODING_RULES.md 通用编码、复杂度、错误处理、UTF-8、标准命令。 保留。 测试触发矩阵迁到 TEST_RULES，保留摘要和命令表。
docs/rules/FRONTEND_RULES.md Vue/TS、组件、状态、路由、UI、浏览器兼容。 保留。 API 和测试细则只保留指针，避免合并版继续膨胀。
docs/rules/TEST_RULES.md 单测、集成、E2E、冒烟、回归、手工验收、测试数据和失败产物。 新增，和 CODING_RULES 命令表互引。 不重复写具体业务用例，只写触发条件和最低验证。
docs/rules/REVIEW_RULES.md Review 角色、严重度、阻塞条件、AI checker、证据质量。 新增，GIT_RULES 只保留 PR 流程。 不规定 Git 分支，不替代 PR 模板。
docs/rules/GIT_RULES.md 分支、commit、PR、worktree、hook、merge-back。 保留。 Review 内容迁出，只保留流程和门禁。
docs/rules/RELEASE.md 版本、tag、发布、回滚、发布门禁。 保留。 发布测试引用 TEST_RULES，不重复测试矩阵。
docs/memory/*.md 长期上下文、架构契约、决策、债务、缺陷。 保留，拆分过长契约。 不写临时实现流水账。

四、推荐的 AI Coding 工作流

阶段 输入 输出 AI Agent 应该做什么 人类开发者确认什么 禁止跳过的检查点
1. Clarify 用户需求、issue、PR、backlog、口头任务、日志。 目标、验收标准、非目标、影响范围、风险初判。 读取入口层，识别是否缺验收或业务规则；不清楚且会改变行为时先问。 确认真实业务目标、验收边界、优先级、不可做事项。 没有验收标准不得直接编码；红区范围未确认不得动手。
2. Spec 澄清结果、现有实现、接口/DB/权限事实、业务规则。 行为规格、接口契约、状态机、错误处理、兼容策略、验收用例。 对照前后端，写出输入输出和边界；涉及 API/DB/权限必须引用证据。 确认业务规则、字段语义、权限边界、兼容窗口。 不得编造接口；不得把 UI 展示模型当后端 DTO；不得跳过错误路径。
3. Plan Spec、风险等级、模块边界、工作区状态。 实施步骤、文件范围、worktree 决策、验证命令、回滚方式。 最小切片，明确绿/黄/红区；评估是否需要独立 worktree 或子 Agent。 确认高风险计划、红区修改、跨仓协调、发布窗口。 脏工作区未识别不得 staging；高风险提交不得自动完成。
4. Task 计划、backlog、依赖关系、验收点。 可独立验收的任务项，每项有 scope、依赖、风险、验证。 拆成可回滚切片；给子 Agent 的任务必须限制文件/模块和停止条件。 确认任务优先级、依赖顺序、是否允许并行。 不得把多个无关验收点混进一个任务或 commit。
5. Execute 任务项、现有代码、测试、规则。 最小 diff、相关测试、必要文档或 memory 更新。 优先复用现有模式；先测后改；不扩大范围；工作中持续核对后端契约。 只在红区、业务规则不明、范围扩大、无法验证时介入确认。 不得误删代码、跳过 mapper、直接透传 DTO、留下 mock/debug/TODO。
6. Verify 成功标准、改动范围、测试矩阵、运行环境。 真实命令结果、浏览器/API/日志证据、未验证项和原因。 按最小有效集执行，失败时定位根因或明确既有阻塞 TD。 确认未验证项是否可接受，关键链路是否需要人工验收。 禁止用“应该没问题”替代验证；缺陷修复必须有回归证据。
7. Review diff、Spec、Plan、验证证据、PR Packet。 阻塞问题、建议项、放行理由、剩余风险。 按 correctness、architecture、security、performance、verification 五轴审查；高风险使用 checker。 最终确认风险接受、合并策略、发布/回滚责任。 Full Packet 的 Red Team 和跨仓证据不得只填空话。
8. Handoff 已完成改动、验证结果、未完成事项、风险。 交付摘要、影响范围、验证、风险、下一步、Memory 判定。 把能继续工作的上下文留下；必要时更新 memory 或 agentmemory。 确认是否进入验收、合并、发布或下一轮任务。 worktree 未 merge-back 不得宣称完成；未说明风险不得交付。

五、优化调整实施计划

第 1 阶段：立即修复，1 到 2 天内完成

具体任务 产出物 验收标准
修复 PR 模板与 validate-pr-packet.mjs 的结构不一致。 更新后的脚本或模板、对应测试 fixture。 Fast Path、Lightweight、Security Full、DB Full、Browser Verification Full 五类 PR body fixture 均通过或按预期失败。
恢复或明确质量基线，收敛阻塞 full lint/typecheck 的 Open TD。 质量基线清单、TD 处理 PR 或临时豁免表。 pnpm lint、pnpm check:type 在主集成分支可稳定运行；如不能，必须有 TD ID 和影响范围。
将红区确认字段标准化。 PR 模板与 AGENTS 小修订。 红区确认包含确认人、时间、范围、原因、验证、回滚六项。

第 2 阶段：规范补齐，3 到 7 天内完成

具体任务 产出物 验收标准
新增 API_RULES.md、DB_RULES.md、TEST_RULES.md、REVIEW_RULES.md。 四份规则文档及索引更新。 AGENTS、QUICKSTART、rules README、PR 模板均能路由到新规则；check:governance 更新并通过。

第 4 阶段：自动化与持续优化

具体任务 产出物 验收标准

六、可直接落地的改进建议

需要新增的规范

  • API_RULES.md：接口契约、DTO/ViewModel、错误码、兼容、mock、跨仓证据。
  • DB_RULES.md：前端仓对 DB/migration/seed/rollback 的跨仓协作协议。
  • TEST_RULES.md：测试层级矩阵、测试数据、失败产物、未验证声明。
  • REVIEW_RULES.md：Review 五轴、严重度、AI checker、证据质量。

需要合并或瘦身的规范

  • FRONTEND_RULES.md 中 API/测试细则改为摘要和链接。
  • GIT_RULES.md 中 Review 内容只保留流程，质量清单迁到 REVIEW_RULES。
  • ARCHITECTURE.md 中模块级长契约逐步迁到模块契约文件。
  • QUICKSTART.md 保留路由表，不承载新增规则正文。

需要强制执行的流程

  • Full Packet 必须有对应证据 section，且 PR 模板和校验器字段一致。
  • API/DB/权限/患者报告变更必须有后端证据，证据最少包含路径、接口、字段、测试或日志。
  • 缺陷修复必须提供回归测试或不可自动化的最小复现和人工验证记录。
  • worktree 任务未 merge-back 到目标分支前不得宣称完成。
  • 质量基线红灯时，交付必须引用 TD ID，不能只写“历史问题”。

可以自动化的检查

  • PR Packet 与模板结构一致性 fixture。
  • backlog ID、任务文件、依赖、状态、验收字段一致性。
  • AGENTS/rules/templates/memory 索引完整性，当前 validate-governance 已有基础，可继续扩展。
  • 声明修饰器后的证据 section 检查，例如 Backend Cross-check 必须出现 cross-repo evidence。
  • 禁止 Full Packet 的 Attack result、Actual results 使用空泛词。

需要人工确认的地方

  • 红区：权限、登录态、Axios 全局层、环境变量、CI/CD、构建发布、全局主题。
  • 业务规则：状态机、统计口径、患者/报告数据、审计、导出、回滚。
  • 高风险提交：跨仓契约、DB 迁移、生产问题、共享组件契约、批量重构。
  • 接受未验证项：E2E 不可运行、后端环境不可用、浏览器验证被登录态阻塞。

适合交给 AI Agent 的地方

  • 入口层阅读、影响面初筛、规则路由、任务确认草稿。
  • 模块内低风险实现、测试补齐、mapper 和格式化逻辑整理。
  • PR Packet 初稿、验证命令汇总、Memory 更新候选判断。
  • 只读审查：检查 diff 是否符合规范、寻找接口/字段/测试遗漏。
  • 治理自动化脚本和 fixture 增补，但红区脚本变更仍需人工确认。

七、最终推荐版本

最适合当前项目的方案不是“大而全流程手册”，而是“一个入口、四个专业补丁、两个自动化闭环”。

最终规范骨架

  • AGENTS.md：继续作为最高入口，只回答读什么、风险、边界、验证、交付、Memory。
  • QUICKSTART.md：维护最小阅读路径，新增 API/DB/TEST/REVIEW 路由。
  • DYNAMIC_WORKFLOW_RULES.md + TASK_LIFECYCLE_RULES.md：保留，前者管风险证据，后者管需求生命周期。
  • API_RULES.md、DB_RULES.md、TEST_RULES.md、REVIEW_RULES.md：补齐中大型项目与 AI 协作的专业缺口。
  • GIT_RULES.md、RELEASE.md：保留流程和发布门禁，引用 TEST/REVIEW 细则。
  • docs/memory/*.md：继续保存长期上下文，但把模块级长契约逐步拆出。

最终工作流

  1. 每个任务先用 AGENTS 五问定位，不清楚则 Clarify。
  2. 涉及接口、DB、权限、患者报告、统计口径时必须进入 Spec，并引用前后端证据。
  3. 多步或高风险任务必须 Plan + Task 拆分，明确 worktree 和验证。
  4. 执行时只做当前验收点，保持最小 diff，测试跟随改动。
  5. 完成前 Verify，真实命令和结果写入交付或 PR。
  6. 高风险、子 Agent 产出、PR 前必须 Review，质量证据由 reviewer 判断。
  7. 交付时写 Summary、Impact、Validation、Risks、Workflow/Memory 判定；需要续接则 Handoff。

最终门禁

  • P0 门禁: PR 模板和校验器必须同源，Full Packet 不允许结构漂移。
  • P0 门禁: 红区无人工确认不得改，范围扩大必须重新确认。
  • P1 门禁: API/DB/权限/患者报告变更无跨仓证据不得交付。
  • P1 门禁: 缺陷修复无回归证据不得标记完成。
  • P1 门禁: 主分支质量基线失败必须有 TD 绑定，不能长期口头豁免。

以这个版本推进后，项目会保留当前治理体系的优点，同时补上最容易被 AI Coding 放大的四个缺口：接口编造、DB 误判、测试不足、审查空心化。这样规范才不只是“看起来完整”，而是能在长期多人、多 Agent、多分支协作中持续降低事故率。
