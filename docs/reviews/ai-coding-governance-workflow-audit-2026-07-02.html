一、第一性原理分析

规范体系的底层目的不是“让文档看起来完整”，而是降低中大型项目中的认知成本、协作成本和回滚成本。对 AI Coding 场景尤其如此：AI Agent 能快速产出代码，也能快速放大错误契约、误删测试、跳过验证、编造接口或把局部假设写成全局事实。

规范要解决的根问题

  • 决策一致性: 不同开发者和不同 AI 工具面对同类任务时，应选择同一读文档路径、风险档位和验证门槛。
  • 变更可追踪: 需求、任务、代码、测试、Review、交付证据必须能串起来，而不是散落在对话和临时文件中。
  • 契约可保护: 前端、后端、数据库、mock、缓存和本地存储之间的字段语义必须可追溯。
  • 失败可回滚: 红区、DB、权限、生产问题不能只有“通过路径”，还必须有失败路径和回滚方案。

当前底层风险

  • 主工作区仍缺 API/DB/TEST/REVIEW 专项规则，导致很多关键要求只能散落在前端或 Git 规则中。
  • 优化分支已补齐规则和脚本，但未合回，团队和 AI 仍会读到旧入口。
  • pnpm 环境门禁阻断标准命令，容易诱发“脚本级验证通过但项目级命令未过”的灰区交付。
  • 任务状态草稿大量存在但未统一纳入可校验状态模型，新人和 AI 难以判断哪些任务可开工、阻塞或已交付。

因此，本项目的治理优先级应从“继续写更多规范”转向“把已补齐的规范合回、校验、嵌入 PR 门禁，并让每个阶段都有可验证产物”。规范只有进入执行链路，才真正支撑长期稳定开发。

七、最终推荐版本

最终建议: 采用“AGENTS 入口 + QUICKSTART 最小阅读 + DYNAMIC_WORKFLOW 风险路由 + API/DB/TEST/REVIEW 专项规则 + PR Packet/治理脚本自动门禁”的组合。当前最关键动作不是继续扩写报告，而是把已验证的治理优化分支合回主工作区。

推荐最小可执行体系

  • AGENTS.md：入口、风险、红区、Memory、交付。
  • QUICKSTART.md：场景最小阅读。
  • DYNAMIC_WORKFLOW_RULES.md：Workflow、Packet、修饰器、Red Team。
  • API_RULES.md / DB_RULES.md / TEST_RULES.md / REVIEW_RULES.md：专业细则。
  • GIT_RULES.md：分支、worktree、PR、merge-back。
  • TASK_MANAGEMENT_RULES.md：backlog 与任务执行单元。
  • RELEASE.md：发布、tag、回滚。

最终门禁

  • P0 门禁: 治理优化分支必须完成 Review、commit、merge-back。
  • P0 门禁: pnpm 标准命令阻塞必须有环境修复或正式技术债绑定。
  • P0 门禁: PR Packet 模板与校验器必须同源。
  • P1 门禁: API/DB/权限/患者报告变更无跨仓证据不得交付。
  • P1 门禁: 缺陷修复无回归证据不得标记完成。
  • P1 门禁: Review 不得只看命令通过，必须检查证据质量和剩余风险。

这套最终版本足够简洁：入口文档告诉 AI 读什么、风险多高、能不能动；专项规则告诉 AI 怎么做；脚本和 PR 模板防止关键字段缺失；人工 Review 负责判断证据是否可信。它能支持 Codex、OpenClaw、Claude Code、Cursor 等 AI Coding 工具在多 worktree、多分支、多 Agent 场景下长期协作。
