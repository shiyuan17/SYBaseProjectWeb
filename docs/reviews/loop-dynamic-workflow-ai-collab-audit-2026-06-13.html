Governance Audit · 2026-06-13

Loop / Dynamic Workflow / AI Coding 协作审计报告

本报告检查 SYBaseProjectWeb 的 Loop Engineering、动态任务路由、AI coding 协作、PR/CI/Hook 门禁、Memory Update 与多 Agent 协作是否顺畅，并对照 OpenAI、GitHub、Anthropic 与 NIST 的公开最佳实践给出风险分级和改进路线。

审计类型: 治理与协作流程 范围: 前端仓库规则与门禁 结论: 成熟可用，仍需提升证据质量 运行时影响: 无

Executive Summary

当前项目的 AI coding 协作体系已经从“文档提醒”进入“可执行治理”阶段： AGENTS.md 负责协作边界与红区确认， DYNAMIC_WORKFLOW_RULES.md 负责 Workflow 与修饰器， LOOP_ENGINEERING_RULES.md 负责循环方式与停止条件， PR 模板、GitHub Actions、lefthook 和治理脚本提供最低机器护栏。

总体判断： 协作链路顺畅度为 8.2 / 10。体系的 “任务确认 - 风险路由 - Loop/Workflow Packet - 验证 - Memory Update” 闭环清晰，已经接近成熟团队的 AI coding operating model。主要剩余风险不在规则缺失， 而在证据质量、人审负担、低风险任务摩擦、以及 Red Team / Checker 的真实执行深度。

Friction Points

等级 问题 影响 建议动作
P0 未发现阻断级治理缺口。 当前体系可继续用于真实交付。 保持现有单一来源边界，不引入第二套路由体系。
P1 PR Packet validator 只能检查字段存在和最低形状。 “Attack result: 已检查无问题”这类低质量证据仍可能绕过机器门禁。 增加 reviewer 检查清单或样例库，明确高质量 Red Team 证据长什么样。
P1 复杂任务协作规则已经写明，但真实采纳/驳回子 Agent 结论的案例较少。 高风险任务可能形式上使用 Checker，实质仍是 maker 自证。 下一个高风险 PR 中强制保留一段 Checker provenance 和采纳记录，沉淀到示例文档。
P1 Triage Loop 缺少周期性输出样例。 规则能指导“如何做”，但团队难以判断一次合格 triage 的颗粒度。 在 docs/templates/ 补一个 Triage 摘要样例，或在下次真实 triage 后归档。
P2 入口文档体系完整但偏重。 新会话容易把时间花在读规范，而不是快速定位本次任务命中的最小上下文。 继续维护 Fast Path 和专项映射表，避免把所有附录重新塞回首读路径。
P2 Memory Update 仍可能被机械填写。 “无 durable context change”若缺少判断依据，reviewer 难以确认是否漏记。 在 PR review 中抽查影响 ARCHITECTURE/DECISIONS 的变更是否真正登记。
P3 浏览器验证证据格式可继续统一。 UI 改动有时会出现截图、录屏、DOM 断言和 E2E 口径不一致。 为 UI Workflow 增加一条“最小浏览器证据”示例即可，不需要新工具。

Recommended Roadmap

短期，1 到 2 个 PR 内

  • 把本报告作为审计基线归档，后续治理改动引用它而不是重复从零审计。
  • 补一个高质量 Red Team / Checker 证据样例，重点展示攻击路径、预期失败点、实际结果、剩余风险。
  • 在下一次高风险任务中真实记录子 Agent 或 Checker 结论的采纳/驳回过程。

中期，1 到 2 个迭代内

  • 为 Triage Loop 增加一个真实输出样例，明确候选任务、去重、风险分类和 State Sink。
  • 建立“PR Packet 证据质量 review checklist”，保持人工判断，不强推复杂自动化。
  • 定期归档关闭的 TECH_DEBT / KNOWN_BUGS / DECISIONS 历史项，防止记忆层膨胀。

长期，按需要启用

  • 如果团队引入自动 dispatcher，必须以现有 Dynamic Workflow 为事实来源，并保留人工红区确认。
  • 如果启用更多 MCP / skills，继续遵循少而高信号、边界清晰、可降级的工具策略。
  • 如果 PR 证据质量成为瓶颈，再考虑轻量语义 lint；在此之前不建议扩大 validator 到主观判断。
