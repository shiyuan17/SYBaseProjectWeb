一、第一性原理分析

项目需要规范，不是因为团队喜欢“写规则”，而是因为中大型软件在多人、多 Agent、多分支、多 worktree 并行时，最容易先失控的不是代码本身，而是决策边界、上下文来源和交付口径。没有统一规则，AI Agent 会编造缺失上下文，人类会在不同阶段使用不同事实源，最终把同一项工作拆成多套互不兼容的解释。

工作流的根本目的，也不是把 Clarify、Spec、Plan、Task、Verify、Review、Handoff 这些阶段“列出来就算完整”，而是确保每一步都能回答五个问题：输入是什么、输出是什么、谁负责、如何验证、失败后如何回滚。只有这样，规范才真正服务项目目标，而不是变成形式化负担。

当前项目的底层风险已经从“规则缺失”转向“规则是否被统一采用”。从工作区现状看，规则体系已明显增强：多 Agent 协作模型、父任务 orchestrator / 子任务 goal、交接强制闭环、失败经验库和治理脚本门禁都已出现。但从主线治理角度看，真正的核心风险仍是三个：

  • 规则存在于工作区，不等于团队已经基于同一基线工作。
  • 工作流闭环已经被定义，不等于存量任务和历史执行方式已经迁移。
  • 治理校验已经通过，不等于整体质量门禁已经稳定收敛。
第一性原理结论: 当前项目已经接近可长期协作的治理结构，但真正阻塞不是继续新增更多规范，而是把已经正确的规则提交、合回、采用，并把存量任务切换到新的最小执行单元模型。

三、推荐的规范体系结构

当前项目不需要重新发明一套新命名体系，关键是保持单一来源并减少重复解释。推荐结构应以现有增强后的规则为准：

文件 职责 不应承担的内容
AGENTS.md 协作入口、风险分层、交付要求、Memory 判定、规则导航。 不重复展开所有专项规则正文。
CODING_RULES.md 编码基线、验证命令、格式、静态检查与最低工程要求。 不承担任务流转与交接规则。
API_RULES.md / DB_RULES.md 前后端契约、DTO / mapper、兼容、迁移、seed、回滚与跨仓证据。 不承担通用 Git / Handoff 流程。
TEST_RULES.md 单测、集成、E2E、冒烟、回归、手工验收触发矩阵。 不替代交付和审查结论。
REVIEW_RULES.md Review 五轴、严重度、阻塞条件、证据质量、AI checker 使用。 不替代实现和验证步骤。
TASK_LIFECYCLE_RULES.md Clarify 到 Handoff 阶段定义、多 Agent 协作模型、收尾强制项。 不重复维护 backlog 字段结构。

GIT_RULES.md / RELEASE.md 分支、worktree、PR、merge-back、发布、回滚。 不承担业务契约解释。

四、推荐的 AI Coding 工作流

推荐工作流为 Clarify -> Spec -> Plan -> Task -> Execute -> Verify -> Review -> Handoff。其核心不是“阶段越多越好”，而是每个阶段都有清楚的输入、输出、责任边界和不可跳过检查点。

阶段 输入 输出 AI Agent 应做什么 人类开发者需确认什么 禁止跳过的检查点
Clarify 需求来源、背景、已有约束、非目标线索。 澄清结论、成功标准、非目标、风险档位。 主 Agent 识别歧义、补全边界、定位需要读取的规则与模块。 需求是否真被理解，是否触发红区或跨仓。 成功标准和非目标未明确前，不得直接编码。
Spec Clarify 结果、现有实现、接口与数据约束。 规格说明、行为边界、契约、验收与回滚方向。 AI 输出可执行规格，并标出跨仓证据需求。 行为是否与业务预期一致，是否接受兼容策略。 涉及 API / DB / 权限时，缺契约证据不得跳过。
Plan Spec、现状代码结构、验证要求。 实现计划、文件边界、验证清单、风险处理。 主 Agent 拆出实施路径与验证方式，选择最小流程档位。 方案是否可接受，风险是否需要额外确认。 没有验证方案、回滚路径或边界说明时不得进入实现。
Task Plan、backlog、任务目录模型。 父任务、子任务、依赖、执行模式、时间盒。 把任务拆成最小可验证、可回滚、5 分钟可判定的 goal 单元。 父任务是否适合作为 orchestrator，子任务是否拆得足够小。 大任务不得直接作为无边界 Goal 执行。
Execute 子任务、文件边界、停止条件、验证命令。 最小改动、实现证据、未解决问题。 实现 Agent 在指定边界内实现，不扩大范围。 红区改动是否需要额外确认。 发现新缺口时只能记录或拆新任务，不能默认扩范围继续做。
Verify 实现结果、测试命令、浏览器或接口证据。 真实命令结果、失败摘要、未验证项与原因。 测试 Agent 或主 Agent 跑最小充分验证并记录事实。 未验证项是否可接受，是否需要补跑更高层验证。 禁止用“应该通过”替代 fresh verification。
Review diff、验证结果、Workflow Packet、跨仓证据。 阻塞项、风险结论、放行或返工建议。 审查 Agent 只读审查，不与实现者角色混淆。 是否接受残余风险，是否需要更高层审查。 中高风险任务不能只有实现者自证通过。
Handoff 实现、验证、审查与当前状态。 交接摘要、已完成事项、未完成事项、Git 状态、Worktree / 分支状态、Merge-back 状态、续接提示词。 主 Agent 汇总真实完成度，并更新需要的 Memory。 是否接受交付，是否仍存在阻塞或待确认项。 未提交 git、未 merge-back、未验证、未 review、blockedReason 未消除时，不得宣称完成。

多 Agent 启用矩阵

档位 默认组合 何时补充
Fast Path 主 Agent 规则或文档变更多时补文档 Agent。
Lightweight 主 Agent + 实现 Agent 共享逻辑、较难验证或风险略高时补测试 Agent 或审查 Agent。
Full 主 Agent + 实现 Agent + 独立审查 命中权限 / 认证 / 敏感数据时加安全 Agent；改规则或契约时加文档 Agent。

五、优化调整实施计划

第 1 阶段：立即修复，1–2 天

具体任务 产出物 验收标准
提交当前治理改动并完成 merge-back 决策 正式 commit、目标分支合流记录 工作区正确规则不再只是本地事实。
刷新 PROJECT_STATE.md 与当前治理阶段 更新后的入口状态文档 新人和 AI 读入口层即可得到当前治理重点。

确认 fresh pnpm lint 失败根因 问题摘要或修复任务 不再停留在“只知道失败，不知道为什么”。

第 2 阶段：规范补齐，3–7 天

具体任务 产出物 验收标准
把多 Agent 协作、失败经验库、交接强制项纳入主线规则 合回后的规则正文与模板 治理脚本通过，主线规则与工作区规则一致。
补充跨仓证据在 Spec / Review / Packet 中的落点 规则补丁或模板增强 API / DB 变更不再只靠口头提醒。
梳理失败经验库维护机制 状态定义、维护规则 经验库可持续增长而不失控。

第 3 阶段：流程固化，1–2 周

具体任务 产出物 验收标准
选择真实历史任务迁移到目录化模型 父任务目录、子任务样板、执行记录 orchestrator / goal 模型在真实任务中可用。
用真实 PR 演练交接强制项和独立审查 一套完整 Handoff 与 Review 样例 实现者不再自证通过，交接信息完整。
验证 loop 防失控规则 复盘记录或失败经验更新 大任务 Goal 不再无限扩范围执行。

第 4 阶段：自动化与持续优化

具体任务 产出物 验收标准
建立治理健康度趋势面板 轻量 dashboard 或审计汇总页 P0 / P1 关闭率、文档漂移、门禁稳定性可观察。
联动 lint / type / governance 质量门禁 标准命令与 CI 证据矩阵 治理通过不再掩盖质量门禁缺口。
继续拆分架构与模块契约文档 专题契约文档与入口索引 新人和 AI 的局部进入成本下降。

六、可直接落地的改进建议

需要新增的规范

  • 失败经验库生命周期机制。
  • 跨仓证据在交付 Packet 中的固定落点。
  • 治理健康度趋势统计口径。

需要合并或收缩的规范

  • 入口文件只保留导航和高层原则，避免与专项规则正文重复。
  • 架构入口文档只保留导航，逐步把模块契约拆到专题文档。

需要强制执行的流程

  • Handoff 必须写未完成事项、Git 状态、Worktree / 分支状态、Merge-back 状态。
  • 中高风险任务必须有独立审查，不允许实现 Agent 自证通过。
  • 大任务必须先拆成父任务 + 5 分钟子任务，再进入 Goal。

可以自动化的检查

  • 治理索引、模板字段、任务目录模型、子任务 timebox 校验。
  • 近期审计报告索引完整性校验。
  • 交接模板关键字段存在性校验。

需要人工确认的环节

  • 权限 / 认证 / 敏感数据 / 导出等红区变更。
  • 跨仓契约变更是否接受兼容策略。
  • worktree 是否可以 merge-back 与何时宣称完成。

适合交给 AI Agent 的工作

  • 任务澄清、规则索引、任务拆解、治理校验、文档模板填充。
  • 低风险实现、最小验证命令整理、失败模式初稿归纳。
  • 只读审查、差距分析、报告生成与持续追踪。

七、最终推荐版本

对当前项目，最适合长期使用的治理方案，不是继续扩写更多规则，而是把现有增强后的结构稳定下来，围绕以下组合执行：

  • 单一来源矩阵，避免同一主题多处定义。
  • 多 Agent 分工 + 风险触发矩阵，让主 Agent、实现、测试、审查、文档、安全各守边界。
  • 目录化父任务 + 5 分钟子任务，把 Goal 从“无限推进”变成“最小可验证执行单元”。
  • 交接强制闭环，明确未完成事项、Git、worktree、merge-back 状态。
  • 失败经验库，把高代价重复错误变成规则与脚本护栏。
  • 治理脚本门禁，确保规则、模板、索引和任务结构持续自洽。
  • worktree / merge-back 完成定义，防止“局部完成”被误报成“主线完成”。
最终结论: 当前项目已经接近“可长期协作的治理结构”，但关键阻塞不在新增更多规则，而在把工作区里的正确规则提交、合回、采用，并把存量任务迁到新的执行模型。
审计基线：当前工作区事实；未提交治理改动已纳入审计，但不视为主线已完成。
