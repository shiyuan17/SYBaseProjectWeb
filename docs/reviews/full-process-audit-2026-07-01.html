一、总体结论

当前系统并非“主流程完全闭环、后端充分支撑”的状态。M2 标本接收、M3 技术标本登记、M4 报告生命周期、M5 归档借阅等主链路已经具备相当一部分真实后端能力，但仍存在多处会影响审计结论的明显缺口：前端页面已开放但后端未落地、关键工作站按钮仍停留在“待接入”、技术任务写操作权限过粗、统一流程事件与统一审计并未覆盖所有关键对象与动作、部分状态设计过粗且缺少撤回/取消/返工语义、部分借阅/库存链路仅靠应用层先查后改，未看到显式锁或版本控制。
结论不建议定性为“可放心上线的完整闭环系统”。尤其是冰冻流程、若干技术医嘱工作站、技术任务权限控制、统一流程追踪完整性这几块，已经属于需要优先整改的高风险区域。

已确认相对完整的主链

  • M2 收样后会创建 pathology_cases，并确保技术登记待办存在。
  • M3 技术标本登记完成后会创建 GROSSING 技术任务。
  • M4 报告草稿、提交、审核、驳回、签发、发布存在真实状态流转与主要流程事件。
  • M5 归档借阅、归还、异常登记具备真实后端与落库。

已确认的重大缺口

  • 冰冻工作台为前端 mock，未发现对应真实后端流程闭环。
  • 多个技术医嘱工作站仅展示查询与按钮壳子，关键动作未接线。
  • 技术任务写操作权限仅依赖 TECHNICAL_TASK_QUERY。
  • 统一审计依赖 @RequirePermission / @AuditOperation，不是全覆盖。

审计口径提醒

  • “前端有入口但后端没有支撑”已单独列出。
  • “后端有接口但前端未明显使用”只列证据足够的点，不做全量臆断。
  • 并发风险如果只看到应用层先查后改、未看到锁/版本字段，会按中高风险记载。

5 审计追踪 正式报告分发 报告打印/发放/回收主要只更新 report_versions.print_status / delivery_status，未见统一 workflow event 记录。 报告发放历史、回收审计、外发追责 高 为打印、即时发放、计划发放、回收分别补流程事件与操作日志正文。
6 权限/审计风格不一致 通知中心、操作人核验 MyNotificationController 与 OperatorVerificationController 未使用统一 @RequirePermission；前者依赖当前登录用户隔离，后者仅依赖当前请求用户上下文。统一审计拦截器因此不会按同一规则接管。 审计口径一致性、敏感接口治理 中 为此类接口明确“仅认证即可”或“需权限码”的统一规则，并补专项审计注解。
7 状态设计 M2 标本状态机 SpecimenStatus 仅有 REGISTERED / VERIFIED / FIXING / FIXED / CHECKED_IN / IN_TRANSIT / RECEIVED / REJECTED / RETURNED，缺少撤回、取消、待接收、返工等中间态。 状态说明、页面展示、异常补偿 中 补细状态或补状态历史表，至少区分“待处理”“处理中”“撤销/取消/退回”。
8 并发一致性 M5 借阅/白片库存 归档借阅与白片借阅能防重复，但主要是“先查状态/库存，再 update”；未看到显式锁、版本字段或数据库侧幂等约束来避免并发脏写。 重复借出、库存负数、状态覆盖 中 对关键库存/借阅记录引入乐观锁版本、条件更新或悲观锁。
9 流程语义断层 M5 材料借阅 material_loans 表含 approved_by_user_id / approved_by_name 字段，但当前控制器/服务未见单独审批动作，审批语义与实际流程不一致。 借阅责任划分、审计解释、流程设计清晰度 中 若不需要审批，删去语义误导字段；若需要审批，补审批节点、状态与权限。
10 接口使用断层 技术登记 API 前端定义了 getTechnicalSpecimenRegistrationDetail(caseId)，但运行时代码未找到非测试引用；实际页面直接走 workspace 接口，存在后端/前端接口冗余风险。 维护成本、接口收敛 低 确认是否保留独立 detail 接口；未用则下线或标记为兼容 API。
11 统一审计覆盖不足 全系统 ApiAuditInterceptor 只审带 @RequirePermission 的请求；非 GET 才默认审计，GET 还要求 @AuditOperation(sensitiveQuery = true)。 敏感查询、接口治理、一致性审计 中 对关键查询补 sensitiveQuery，并梳理“不使用权限注解”的控制器如何纳入统一审计。
12 前后端实现不对称 技术流程/工作站 少数工作站（如常规医嘱）已接入真实动作，而同类工作站仍停留在壳层，形成“同模块能力成熟度明显不一致”的用户感知断层。 操作培训、流程一致性、上线认知 中 按流程优先级统一成熟度，避免同菜单下部分可用、部分假可用。

三、流程缺失检查

3.1 申请单 / 标本 / M2 到 技术登记 / M3

当前已实现的流程节点

  • 申请单创建、编辑、作废存在真实 service 与 repository 落库。
  • 标本确认、固定、收样、接收、运输存在真实控制器与状态推进。
  • 接收环节 SpecimenReceiptAndRemovalService.processReceipt() 会创建病例，并执行 ensureTechnicalSpecimenRegistrationPending(applicationId, caseId)。
  • 技术标本登记工作台存在真实接口：pending/list/detail/workspace/application-workbench/materials/verify/cancel-verification/detail-sections/media-assets/complete。
  • 技术登记完成后，TechnicalSpecimenRegistrationService 会调用完成登记并创建 GROSSING 任务。

缺失的流程节点

  • 申请单主对象未见纳入统一 workflow_events 主线的创建/编辑/作废事件。
  • 未见单独的“撤回申请单”“恢复作废申请单”“收样后回退到申请单层处理”的完整闭环。
  • 标本状态机未显式表达“待接收”“已接收待登记”“取消运输”等更细状态，很多语义依赖其他表或时间字段侧推。

不合理的流转点

  • 申请单对象和标本对象的生命周期追踪不在同一条统一事件链上，跨对象回放时容易断裂。
  • 主状态过粗，容易出现前端展示写成“已处理”，但实际只是“进入下一个工作台待办”的解释偏差。

应补充内容

  • 补 APPLICATION_CREATE / UPDATE / VOID 流程事件。
  • 补申请单撤回/恢复/退回的业务判定与状态表达，若业务不支持则需明确禁止并写入规则。
  • 为标本主状态增加中间态或历史状态表。

3.2 技术处理 / M3 到 诊断报告 / M4

当前已实现的流程节点

  • 技术标本登记、取材、脱水、包埋、切片、染色、返工、任务池均有真实后端控制器与 service。
  • 报告流程存在草稿、保存、提交、审核、驳回、签发、发布的主线状态推进。
  • 报告签发/发布会同步更新病例状态，并在发布时触发 billing。

缺失的流程节点

  • 冰冻工作台未形成真实后端闭环，不能视作正式流程节点。
  • 若干医嘱工作站（特检/IHC/细胞学/液基）页面存在，但关键动作未接入。
  • 正式报告分发（打印/发放/回收）未进入统一 workflow event，导致“报告内容完成”与“报告实际交付”两条线分离。

不合理的流转点

  • 技术任务的写操作权限与查询权限未分离，业务上相当于“能看任务池的人大概率也能改任务池”。
  • 报告驳回会把诊断任务退回进行中，但正式版本的分发历史与主流程事件链割裂。

应补充内容

  • 冻结未落地的冰冻流程入口，或补全真实任务/状态/数据库/接口。
  • 为所有技术医嘱工作站补齐真实动作链。
  • 为正式报告打印/发放/计划发放/回收补流程事件、幂等保护与异常补偿。

3.3 归档 / 借阅 / M5

当前已实现的流程节点

  • 归档柜、归档位、批量归档、借阅、归还、异常登记、医疗废物、白片借记均有真实后端表与接口。
  • V22__create_m5_archive_operation_schema.sql 已建立 archive_positions、specimen_storage_records、material_loans 等核心表。
  • 归档借阅会更新存储记录状态并释放原归档位，归还时会重新占用归档位。

缺失的流程节点

  • material_loans 表面上存在审批人字段，但未见单独审批动作、审批状态或审批页面。
  • 未见明确“借阅撤销”“归还撤销”“借阅审批驳回”的状态节点与接口。
  • 白片借记只有借出/归还，没有看到单独异常登记或库存纠偏流程。

不合理的流转点

  • 借阅与库存链路以应用层判断为主，未见显式并发锁或版本控制。
  • 借阅审批语义停留在表字段层，没有形成业务闭环，容易误导后续开发和审计解释。

应补充内容

  • 明确材料借阅是否需要审批；需要则补审批状态、动作、权限、日志，不需要则删冗余字段。
  • 给归档借阅与白片借记补条件更新/版本号/锁策略。
  • 给高风险库存操作补失败补偿与冲突提示规范。

3.4 前端有入口但后端没有支撑

已确认的典型问题只有一项，而且足够严重：冰冻工作台。前端真实路由已开放，页面会调用 #/modules/frozen-workflow/api/frozen-workflow-service，但服务端分支恒定走 mock，真实分支直接抛错；后端未找到与之匹配的真实流程控制器依据。

3.5 后端有接口但前端未明显使用

  • getTechnicalSpecimenRegistrationDetail(caseId)：前端 API 有定义，但运行时代码未找到非测试引用；页面实际上主要走 workspace 接口。
  • 未继续穷举全仓所有“后端有接口但前端未用”的情况；本报告只写证据足够的例子，其他不做武断结论。

四、状态机检查

4.1 标本主状态机（当前代码表达）

问题判断

  • SpecimenStatus 本身缺少 CANCELLED / WITHDRAWN / RECEIPT_PENDING / REWORK 一类中间态。
  • FixationStatus 只有 PENDING / FIXING / COMPLETED / ABNORMAL，表达能力偏粗。
  • 部分异常动作只能通过外围记录表或时间字段推断，主状态机自身不可读。
  • 未在已审代码中看到一个独立、统一的“非法跳转状态机守卫类”；更多是分散在 service 内按动作判断。

问题判断

  • 报告主状态链比较清晰，且 DiagnosticReportLifecycleService 对提交、审核、签发、发布前置状态均有后端校验。
  • 但“报告内容生命周期”和“报告交付/分发生命周期”是两套状态，当前统一追踪没有把两套状态完整收敛到一个事件视图中。
  • 打印/发放/回收缺少统一 workflow event，因此运维和审计在回放时只能看到版本字段变化，不能看到标准化操作节点。

4.3 技术任务状态与权限问题

  • TechnicalTaskManagementService.requireTask() 仅阻止对已完成任务继续写入，但未见按角色/岗位对“谁可以 assign / claim / release / priority / remarks”做细分控制。
  • 也就是说，状态校验存在，权限边界却过粗，这不是状态机正确就能弥补的问题。
  • 任务写操作会补 workflow event 与通知，但如果写权限本身过宽，日志只会记录“谁改了”，不能阻止“本不该改的人也能改”。

4.4 缺少操作日志/审计日志的状态动作

已确认缺口：申请单 create/update/void 未见统一 workflow event；正式报告 print/issue/recall 未见统一 workflow event；未使用 @RequirePermission 的控制器不会进入 ApiAuditInterceptor 这一套统一拦截审计。

5.2 数据可能丢失或断链的节点

  • 申请单对象的统一流程事件未找到依据，导致主对象在统一追踪中断链。
  • 正式报告分发/回收只写版本分发字段，不写标准流程事件，导致“报告已发布”之后的对外交付历史不在统一事件链中。
  • 冰冻流程当前为 mock，前端交互不会形成真实后端数据与状态链。

5.3 前后端字段/含义可能不一致的点

  • 技术登记前端保留了 detail 和 workspace 两套接口模型，但运行时主要走 workspace，接口职责已经出现重叠。
  • 报告“主状态”和“分发状态”是两套字段，前端若用单一状态标签概括，容易误导业务人员把“已签发/已发布”和“已打印/已发放”混为一谈。
  • material_loans 表含审批字段，但前端/后端当前流程语义看起来更像“直接借出”，表意与流程不一致。

5.4 只更新主表、不更新关联表的风险点

  • 正式报告打印/发放/回收只更新 report_versions，未同步补写统一事件表。
  • 白片借记更新库存与借据表是同事务，但未见独立库存事件表；异常纠偏时回放粒度不足。
  • 统一操作日志和流程事件不是所有动作都双写，有些动作只进一个体系，审计视图容易不一致。

5.5 缺少事务保证或并发控制的点

多数关键 service 已标注 @Transactional，但事务并不等于并发安全。归档借阅和白片借记虽然有事务，但当前看到的实现仍是“先查状态/库存，再 update”，未见显式 FOR UPDATE、版本号字段、条件更新返回值校验等并发防线。高并发下仍有重复借出、库存覆盖、最后写入覆盖前写入的风险。

六、接口与数据库修复建议

6.1 需要新增或调整的接口

  • 补冰冻流程真实接口：待处理列表、接单/开始、过程记录、完成、退回、超时、诊断结果回填、追踪查询。
  • 为特检/IHC/细胞学/液基工作站补动作接口映射，至少覆盖新增、确认、完成、退回、取消。
  • 若 M5 借阅需要审批，新增借阅提交、审批通过、审批驳回、撤销申请接口；若不需要审批，删去审批字段并同步精简 VO/DTO。
  • 补统一流程追踪读取接口，使申请单事件与报告分发事件也能进入同一时间线。

6.2 需要新增或调整的数据库字段/表

  • 为申请单补统一流程事件记录，优先复用 workflow_events；若模型不兼容，则新增对象级状态历史表。
  • 为报告分发动作增加标准事件记录，至少包含版本 ID、动作、操作人、操作时间、终端、备注。
  • 为白片库存或借阅表增加版本字段，或在更新 SQL 中使用条件更新保证并发安全。
  • 重新评估 material_loans.approved_by_* 字段：要么补齐审批状态字段，要么删掉避免语义污染。

6.3 需要补充的状态校验

  • 技术任务写操作除“未完成”外，还应校验当前操作者是否具备对应写权限、是否属于允许岗位、是否允许修改他人已认领任务。
  • 标本状态需补撤回/取消/返工相关的显式后端规则，不应完全依赖“当前 service 有没有走到那条分支”。
  • 报告分发需要明确区分“未打印不可发放”“已发放才可回收”“计划发放不可重复计划”。

6.4 需要补充的事务处理

  • 对归档借阅、归还、白片借记、白片归还增加并发安全策略：悲观锁、乐观锁或条件更新。
  • 对“状态变更 + 日志事件 + 通知/任务生成”这类复合动作，失败时需明确回滚边界，避免主表成功、事件失败。
  • 对批量报告分发操作补部分成功场景的审计细粒度记录。

6.5 需要补充的操作日志/审计日志

  • 申请单 create/update/void 写统一流程事件。
  • 正式报告 print/issue/schedule/recall 写统一流程事件。
  • 为未使用 @RequirePermission 的敏感接口显式补 @AuditOperation 或主动 OperationAuditService。
  • 提高统一操作日志正文可读性，不只记录 HTTP method/uri，还应记录关键业务对象与动作摘要。

6.6 需要补充的权限控制

  • 拆分 TECHNICAL_TASK_QUERY 为查询、分派、领取、释放、调优先级、改备注等至少 5 类权限。
  • 明确 OperatorVerificationController 的访问规则：是任何登录用户可调用，还是只允许特定流程节点发起。
  • 统一通知中心与个人偏好接口的认证/授权设计说明，避免审计上出现“没有权限注解是否默认开放”的歧义。

七、可执行修复计划

P0：必须立即修复

  • 关闭或隐藏冰冻工作台入口，或者同步补完真实后端闭环；在补完前不能把该流程算作正式可用能力。
  • 修复技术任务池写操作权限过粗问题，至少禁止只有查询权限的用户执行分派、领取、释放、改优先级、改备注。
  • 对“页面已展示但动作待接入”的技术医嘱工作站做下线或能力标识收缩，避免用户误操作和流程断点。
  • 补申请单主对象与正式报告分发动作的统一事件记录，先把主追踪链补齐。

P1：重要修复

  • 为归档借阅、白片借记补并发安全机制，防止重复借出、库存覆盖。
  • 统一未使用 @RequirePermission 的控制器治理方式，并补统一审计。
  • 梳理 M2 标本主状态与异常流程，补足撤回/取消/返工等状态表达。
  • 明确 M5 借阅是否存在审批节点，消除表结构和业务流程语义不一致。

P2：优化项

  • 收敛重复或未使用的接口，如技术登记 detail/workspace 双轨问题。
  • 统一“主状态”和“分发状态”的前端展示规范，避免用户误读。
  • 提高操作日志正文质量，强化业务对象可追踪性与回放效率。
  • 对批量动作补幂等设计与回执结构标准化。
