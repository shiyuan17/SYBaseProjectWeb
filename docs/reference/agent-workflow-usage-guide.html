AI 工作流使用教程

这份文档教你如何把任务交代给 AI：先澄清目标和验收标准，再选择合适的 Workflow 和技能组合，执行中看风险，交付时看证据。它不是新的项目规范， 而是一份日常使用手册。

你负责 目标、边界、验收标准、风险确认
AI 负责 探索、计划、实现、验证、交付说明
核心原则 先澄清，再执行；先验证，再交付

一、你什么时候需要这套工作流

只要任务需要 AI 读代码、改代码、写文档、排查问题或给出交付判断， 就应该使用工作流。越是跨模块、跨仓、权限、生产问题，越不能只用一句 “帮我改一下”开工。

Step 1
1. 说清目标 告诉 AI 你要解决什么问题，而不是只描述想改哪里。
2. 给验收标准 说明什么现象算完成，哪些行为不能改变。
3. 判断风险 识别是否涉及权限、接口、构建、生产问题或跨仓联动。
4. 选 Workflow 让 AI 声明主 Workflow、修饰器和关键假设。
5. 配技能 按任务类型选择少量有用 skill，不叠仪式感。
6. 看证据 交付时只接受真实验证结果和清楚风险说明。

小改动

例如文案、局部样式、单个测试修复。仍要说清验收标准，但可以用轻量流程。

低风险 最小验证

中型功能

例如新增页面、联调接口、调整状态流。先澄清和计划，再让 AI 实现。

中风险 需要计划

高风险任务

权限、登录态、生产故障、构建发布、全局请求层等，必须先确认风险和边界。

高风险 先确认

二、任务开始前怎么让 AI 澄清

好的任务开头会节省最多返工。你可以先要求 AI 使用任务澄清思路， 把目标、范围、非目标、验收标准和风险讲清楚，再进入实现。

Step 2
默认话术： 先使用任务澄清 / brainstorming 思路，确认目标、影响范围、 验收标准、非目标和风险后再动手。如果需求存在多种理解，先不要编码，先问我关键问题。

什么时候用 brainstorming

  • 你还没完全确定页面行为、数据流或验收标准。
  • 这个任务可能有 2 到 3 种实现方向。
  • 涉及 UI、接口、状态、路由或权限组合变更。
  • 你希望 AI 先提出方案，而不是直接改代码。

什么时候用 grill-me / grill-with-docs

  • 你已经有方案，但担心漏了风险。
  • 你希望 AI 追问你验收标准和边界。
  • 方案必须对照仓库规范、现有实现或历史决策。
  • 需求来自口头描述，需要被拆成可执行任务。
可复制提示词：小任务
请先确认这个任务是否属于绿区小改。
任务目标：[写清楚要改什么]
验收标准：[写清楚看到什么算完成]
请先输出任务确认、影响范围、主 Workflow 和验证方式。
如果没有歧义，再直接做最小改动。
可复制提示词：中型功能
先使用 brainstorming 思路做任务澄清，不要直接编码。
请确认目标、非目标、影响范围、数据流、验收标准和风险。
如果存在多种实现方式，请给出 2 到 3 个方案并推荐一个。
确认后再进入实现计划和代码修改。
可复制提示词：高风险任务
这个任务可能涉及红区或高风险。
请先不要修改代码，先识别是否涉及权限、登录态、请求层、构建发布、患者/报告数据、跨仓接口或生产问题。
请输出必须人工确认的点、推荐 Workflow、需要启用的 skill、验证证据清单。
等我确认后再执行。

三、做不同任务时怎么选 Workflow

Workflow 是任务的“打法”。你不需要记所有细节，只要能判断任务属于哪一类， 然后要求 AI 按对应打法提供证据。

Step 3

UI Workflow

页面、组件、布局、交互、图表、移动端或浏览器兼容。

你要先确认

目标用户、页面状态、空态/加载/错误态、桌面和移动视口。

要求 AI 做

使用 UI 专项 skill，真实浏览器验证，给截图或浏览器检查结论。

API Workflow

请求模型、字段映射、分页、错误码、mock、联调。

你要先确认

字段含义、成功/失败响应、分页边界、是否需要后端核对。

要求 AI 做

使用 API 设计 skill，对照后端实现，覆盖成功和失败分支。

DB Workflow

后端迁移、种子、SQL、历史数据兼容、回滚。

你要先确认

目标环境、旧数据、新数据、回滚路径和后端验证证据。

要求 AI 做

前端只验证展示和接口兼容，数据库证据必须来自后端。

Security Workflow

权限、登录态、菜单鉴权、患者信息、报告信息、导出、审计。

你要先确认

角色、权限矩阵、敏感数据边界、是否涉及红区。

要求 AI 做

启用安全检查和 Red Team，验证低权限、未登录、直达 URL 等场景。

Architecture Workflow

共享组件、store、路由、request 边界、跨模块复用、重构。

你要先确认

影响面、调用方、兼容要求、是否真的需要抽象。

要求 AI 做

优先查调用关系，先写计划，再改动；交付要说明兼容性。

Production Debug

线上故障、现场阻塞、性能回退、日志里已有错误。

你要先确认

原始现象、发生时间、日志位置、可复现输入、回滚要求。

要求 AI 做

先读日志、先复现、建立反馈环，再做最小修复和回归验证。

Workflow-Infra

CI、hook、构建脚本、包管理、环境变量、发布路径。

你要先确认

是否影响提交、推送、构建、发布或不同操作系统。

要求 AI 做

说明跨平台和回滚风险；涉及红区时先暂停确认。

纯文档任务

教程、说明、注释、信息整理，不改变运行时行为。

你要先确认

受众、交付形式、是否需要导航入口、是否更新记忆层。

要求 AI 做

可标注 Workflow 不适用，但仍要验证格式、链接和交付边界。

四、技能怎么搭配

技能不是越多越好。人工使用时只需要记住一个搭配规则： 一次任务最多选择 1 个流程 skill、1 个专项 skill、1 个验证 skill。

Step 4
任务状态 建议技能 你应该怎么要求 AI
需求不清 brainstorming 或 grill-with-docs 先问清目标、验收标准、非目标和风险，不要直接编码。
需要实现 writing-plans，必要时 test-driven-development 先写实现计划；新增逻辑或修 bug 时优先补测试。
Bug / 故障 systematic-debugging 或 diagnose 先复现和读日志，再定位根因，最后验证原始问题消失。
UI 页面 frontend-ui-engineering + Browser 验证 检查真实渲染、响应式、长文本、空态、错误态和交互。
API 联调 api-and-interface-design + Backend Cross-check 对照后端字段、错误码、分页和兼容分支。
任务来源卡片 using-git-worktrees + 任务起始模板 先核对任务来源、验收标准和关联资料，再为单个任务项 使用独立 worktree。
安全权限 security-and-hardening + Red Team 检查越权、敏感数据泄露、直达 URL、日志和导出。
架构重构 code-simplification 或 code-review-and-quality 先查影响面，避免浅封装和跨层耦合。
交付前 verification-before-completion 要求 AI 用真实命令输出证明完成，不接受推测。

不要这样用

“把所有 skills 都用上，越严格越好。”这会让任务变慢，结论重复， 也容易让 AI 忘记真正的验收标准。

推荐这样用

“这个是 UI 联调任务，请先澄清验收标准，用 UI 专项 skill， 最后做浏览器验证并回填截图/结果。”

六、执行过程中用户怎么检查

AI 开始执行后，用户不需要盯每一行代码，但要盯关键行为： 有没有先读上下文、有没有声明风险、有没有在红区停下来。

Step 6

你应该看到 AI 做这些事

先读相关规范、现有实现、日志或后端代码。
输出任务确认、主 Workflow、风险等级和关键假设。
任务来源先核对编号、 标题、状态、验收标准和关联资料。
说明影响范围，尤其是共享组件、状态、路由、请求层。
遇到红区或不确定业务规则时主动暂停确认。
每隔一段时间说明正在查什么、学到了什么。

你不应该放过这些信号

还没读日志就开始修生产问题。
API 任务只看前端 mock，不核对后端契约。
UI 任务没有浏览器验证或真实截图结论。
权限、登录态、拦截器、构建发布改动没有人工确认。
命中隔离条件的任务直接在主工作区开改，未使用独立 worktree。
为了让检查通过而删除测试、降级或兼容逻辑。

七、交付时用户怎么验收

交付不是“代码改完”。你要看 AI 是否交代清楚做了什么、影响了什么、 怎么验证、还有什么风险。

Step 7

交付必须包含

  • 变更摘要：做了什么，为什么这样做。
  • 影响说明：是否涉及接口、状态、路由、主题、兼容性。
  • Workflow Packet：主 Workflow、修饰器、专家 Agent、动态测试。
  • 验证结果：真实执行的命令和结论。
  • 任务来源：说明对应分支、worktree、PR 关联和剩余待办。
  • Memory Update：更新了什么，未更新原因是什么。
  • 风险提示：还需要人工跟进什么。

验证结果怎么读

  • 看到具体命令，例如 pnpm lint、pnpm check:type。
  • 看到通过、失败或未验证的明确结论。
  • 失败后应有修复和重跑记录。
  • 未运行的项必须说明原因，不能默认省略。
  • 高风险任务要看到 Red Team 或模拟场景结论。

不应接受的交付

“已优化，应该没问题。”这没有说明影响范围、验证命令、失败项、风险和记忆层。 用户无法判断任务是否真的完成。

可以接受的交付

“新增了某页面教程 HTML；未改导航和运行时代码；`git diff --check` 通过； 浏览器桌面/移动视口检查通过；Memory Update 未更新，因为未改变长期规则。”

八、常见错误

如果你在任务开头或验收时发现这些情况，直接让 AI 回到对应步骤补齐。

Step 8

任务描述错误

  • 只说“帮我优化一下”，没有目标和验收标准。
  • 任务来源只给编号，没有标题、背景、验收标准或当前状态。
  • 没有说明哪些行为不能改变。
  • 把高风险任务当普通小改处理。
  • 没有说清楚是否要更新文档、测试或记忆层。

执行过程错误

  • UI 改动没有浏览器验证。
  • API 改动没有对照后端。
  • 生产问题未读日志就修。
  • 权限、登录态、拦截器改动未人工确认。
  • 多个无关任务项共用同一个 worktree。

技能使用错误

  • 一次任务堆太多 skills，导致流程比任务本身更重。
  • 同时使用两个同类 TDD 或调试流程。
  • 让外部 skill 覆盖项目已有规则。
  • 忘记交付前使用验证型 skill 检查证据。

交付验收错误

  • 接受“应该可以”但没有真实命令输出。
  • 任务交付时没有说明来源状态、PR 关联和剩余待办。
  • 没有说明未验证项。
  • 高风险任务没有 Red Team 或角色模拟。
  • 长期决策、债务或缺陷没有检查 Memory Update。

九、任务来源与协作任务

任务平台不是单独的开发 Workflow，而是任务来源和协作载体。用户要做的事是把来源 信息、验收标准、工作区隔离和 PR 对齐讲清楚；AI 再根据内容选择 UI、 API、Security、Architecture 或其他 Workflow。

Step 9

什么时候按任务来源启动

  • 任务来自 issue、工单、卡片、PR 或文档需求，且需要在来源中追踪状态、讨论、PR 和验收。
  • 一个任务需要独立分支、独立验证记录，后续要进入 PR 评审。
  • 多个任务并行时，需要避免不同任务来源的改动混在同一个工作区。
  • 任务要交接给其他人，需要保留 来源、分支、验证和剩余风险的对应关系。

你需要先提供什么

  • 来源编号和标题，例如 ENG-123 用户管理列表筛选。
  • 背景、目标和验收标准；没有验收标准时要让 AI 先澄清。
  • 关联资料，例如设计稿、接口说明、后端讨论结论或历史 PR。
  • 是否允许 AI 更新来源状态、评论、计划文档或 PR 描述。

启动 AI 是否核对来源编号、标题、状态、验收标准和关联资料。 “先补齐任务来源起始信息，再输出任务确认。”
隔离 AI 是否说明独立 worktree、分支名和当前工作区是否有无关改动。 “请先隔离工作区，不要在主工作区混改这个任务。”
规划 AI 是否从验收标准推导 Workflow、skills、测试和风险，而不是只照来源标题做。 “请把每条验收标准映射到实现步骤和验证方式。”
执行 AI 是否持续说明当前进度、遇到的风险、需要人工确认的红区。 “请暂停编码，先说明风险、影响范围和需要我确认的点。”
交付 AI 是否说明来源状态、PR 关联、验证证据、剩余待办和未验证项。 “请按验收标准逐条回填完成证据和剩余风险。”

不应这样启动

“做一下 ENG-123。”这会让 AI 只按标题猜需求，也看不出是否要隔离工作区、 是否有验收标准、是否需要更新来源或 PR。

推荐这样启动

“请按任务来源启动 ENG-123，先核对描述和验收标准；不清楚先问我。 如果风险命中隔离条件， 使用独立 worktree，并把每条验收标准映射到验证方式。”

续接历史任务

先让 AI 读取来源最新状态和已有讨论。
要求 AI 检查当前分支、worktree、未提交改动。
让 AI 输出“已完成、进行中、待处理”。

并行任务

一个实现任务项对应一个 worktree 和一个分支。
多个子 Agent 只能做互不依赖的子任务。
主 Agent 汇总后再统一验证和交付。

PR 对齐

PR 说明要能对应任务目标和验收标准。
UI、接口、安全任务要附对应验证证据。
剩余风险、未验证项和 Memory Update 要写清。

十、一页速查

下面这张清单适合打印或在 PR 前快速扫一遍。只要其中一项答不上来， 就让 AI 回到对应步骤补证据。

Check

任务开始前

目标清楚吗？
验收标准清楚吗？
任务来源编号、标题、状态清楚吗？
影响范围清楚吗？
非目标说清楚了吗？

选择 Workflow

主 Workflow 已声明？
强制修饰器已声明？
红区是否暂停确认？
跨仓是否需要核对？

选择 skills

是否只选了必要 skill？
需求不清是否先澄清？
Bug 是否先诊断？
交付前是否验证？

执行中检查

是否读了相关上下文？
任务是否按风险完成了 worktree 决策？
是否说明正在查什么？
是否避免无关重构？
是否保留兼容逻辑？

交付前检查

变更摘要完整？
验证命令真实执行？
未验证项已说明？
剩余风险已说明？

PR 前检查

Workflow Packet 有了吗？
PR 是否关联了对应任务来源？
Memory Update 说明了吗？
截图/接口证据齐了吗？
风险事项有人跟进吗？
